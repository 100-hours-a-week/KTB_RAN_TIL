### Simple Storage Service (S3)

→ 의미 : **간단**하게 **저장소**를 **제공**합니다.

즉, 스토리지다**. S3 = 스토리지**

**객체 스토리지(Object Storage)**

**: 파일 보관용 창고**

- 이미지, HTML, JS 같은 **파일 저장**이 목적

---

그냥 **파일을 저장하고, 요청하면 그대로 꺼내주는 서비스**야.

서버가 아니고, 계산도 안 한다.

S3는

- 서버 실행 ❌
- 코드 실행 ❌
- 요청 처리 로직 ❌
- **파일 전달 ⭕**

---

### 정적 배포

**미리 만들어진 파일을 그대로 사용자에게 내려주는 방식**

1. 브라우저가 요청함 : “**/index.html** 주세요”
2. S3가 그냥 파일 하나 내려줌
3. 끝.

중간에 **계산도 없고, 조건문도 없고, DB도 없음.**

이걸 **정적(static)이라고** 한다.

---

### 동적 배포

동적 서버(백엔드) = EC2

1. 요청 옴

   → 로그인 확인

   → DB 조회

   → 계산

   → 결과 만들어서 응답


**계산이 필요함.**

---

### 프론트엔드가 정적인 이유

- React, Vue 같은 프론트는 개발할 때는 코드지만 **빌드하면 결과물이 그냥 파일**

  빌드 결과물 예:

    - index.html
    - main.js (JS 번들)
        - JS 번들
            - 개발중에는 여러개의 js 파일이 있지만,
            - 배포를 위한 빌드를 진행하면 여러 파일이 하나로 묶인다.
            - 그 결과물이 main.js, app.js, index.[].js

          → 이것이 js 번들

    - style.css

계산이 필요없는 결과물 3개를 준다. EC2가 필요하지 않다!!

---

### JS는 어떻게 실행되는가?

JS는 사용자의 브라우저에서 실행된다.

1. 브라우저가 S3에 요청 “index.html 주세요”
2. S3가 index.html 파일을 보낸다.
3. 브라우저가 html을 읽음
4. 읽다가 </script src=”main.js”>를 읽음
5. 브라우저가 다시 S3에 요청 “main.js 주세요”
6. S3가 main.js 파일을 보냄
7. 브라우저가 js를 실행.
8. 끝

**→ 결국 S3는 던지고 브라우저가 실행한다.**

---

### JS에서 API 요청 시

위와 연결

1. 브라우저에서 js 실행
2. fetch api 문장 읽음
3. POST /login 실행
4. 브라우저가 EC2에 요청을 보냄
5. EC2는 계산을 통해 값을 보냄
6. 브라우저가 값을 받음.

**→ S3는 어디에도 관여 하지 않는다.**

---

### **가장 표준적인 구조**

- 프론트엔드 → S3 (정적 파일)
- 백엔드 → EC2 (계산하는 서버)

---

### 요약

> S3는 코드를 실행하는 서버가 아니라,
>
>
> **HTML·JS·이미지 같은 정적 파일을 저장하고 그대로 제공하는 스토리지**이며,
>
> 이런 방식으로 프론트엔드를 제공하는 것을 **정적 배포**라고 한다.

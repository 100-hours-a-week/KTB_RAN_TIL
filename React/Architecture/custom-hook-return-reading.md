# 커스텀 훅 반환값을 빠르게 읽는 법 (useRetroSection 사례)

프로젝트 하면서 느낀 건, 이런 구조는 결국 "이 훅이 뭘 반환하는지"를 한 번에 읽는 능력이 핵심이라는 점이다.

---

## 1) 빠르게 이해하는 1단계 사고법

```ts
const {
  activeSection,
  setActiveSection,
  retros,
  isMyPage,
  isLoading,
  isError,
  isFetching,
  hasMore,
  loadMore,
} = useRetroSection();
```

### Step 1: 이름부터 본다

`useRetroSection`

- `use` -> 커스텀 훅
- `Retro` -> 회고 도메인
- `Section` -> 섹션 단위 상태 관리

여기서 1차 판단은 이렇게 한다.

> 회고 페이지에서 섹션 단위로 필요한 상태를 묶은 훅이구나.

---

## 2) 반환값은 카테고리로 나눠 읽는다

한 줄씩 순서대로 읽으면 느리다.
반환값을 묶어서 읽어야 훅의 책임이 빨리 보인다.

### 2-1. UI 상태 제어

```ts
activeSection
setActiveSection
```

탭 전환 같은 UI 상태 제어이다.

### 2-2. 데이터

```ts
retros
```

실제 회고 리스트 데이터이다.

### 2-3. 페이지 맥락

```ts
isMyPage
```

내 페이지 여부를 판단하는 플래그이다.
권한 분기나 버튼 노출 조건에 쓰인다.

### 2-4. 비동기 상태

```ts
isLoading
isError
isFetching
```

쿼리 실행 상태를 나타내는 값이다.
이 조합이면 TanStack Query 기반일 가능성이 높다.

### 2-5. 무한 스크롤 제어

```ts
hasMore
loadMore
```

다음 페이지 존재 여부와 추가 로드 트리거이다.
Infinite Query 패턴과 연결되는 경우가 많다.

---

## 3) 이 훅을 한 문장으로 정의해본다

> 회고 섹션 페이지에 필요한 상태, 데이터, 페이징 제어를 한 곳에서 오케스트레이션하는 훅이다.

이 한 문장이 나오면, 세부 구현을 보기 전에도 훅의 역할 경계가 잡힌다.

---

## 4) 바로 이해가 안 되는 이유

처음 막히는 지점은 보통 같다.

> 변수 이름은 보이는데 값이 어디서 만들어지는지는 아직 안 보인다.

이건 자연스러운 단계이다.

---

## 5) 이해 속도를 올리는 실전 순서

### 방법 1: 선언부보다 반환부를 먼저 본다

훅 파일에서 먼저 `return { ... }`를 본다.
여기서 훅이 외부에 공개하는 책임 범위를 먼저 읽는다.

### 방법 2: 내부 구조를 가설로 세운다

보통 아래 형태를 가진다.

```ts
const [activeSection, setActiveSection] = useState(...)

const { data, isLoading, isFetching } = useInfiniteQuery(...)

const loadMore = () => ...
```

즉, UI 상태 + Query 상태를 합쳐서 노출하는 구조이다.
나도 이 패턴으로 묶어서 보니까 파일 길이가 길어도 훨씬 빨리 읽혔다.

---

## 6) 설계를 읽는 관점으로 마무리

이 훅을 볼 때는 구현보다 먼저 역할을 분류한다.

- View 전용 훅인가
- Feature 레벨 훅인가
- Entity 레벨 훅인가
- 오케스트레이션 훅인가

이 케이스는 "회고 메인 페이지를 렌더링하기 위한 상태 집합을 제공하는 오케스트레이션 훅"으로 보는 게 맞다.
컴포넌트는 이 훅이 주는 값만 소비하고, 페이지 로직은 훅 내부에 모인다.

---

## 정리

빠르게 읽는 공식은 아래 4단계이다.

1. 훅 이름으로 책임을 먼저 추측한다.
2. 반환값을 카테고리로 묶는다.
3. Query 상태와 UI 상태를 분리해서 본다.
4. 한 문장으로 훅의 정체를 정의한다.

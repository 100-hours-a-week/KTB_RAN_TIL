좋아, 이제 **개념은 충분히 잡혔으니까**
아래에서는 **Next.js App Router 기준으로 “진짜 최소 + 이해 쉬운” Service Worker 등록 예시**를 보여줄게.
(불필요한 옵션, PWA 라이브러리 전부 제외)

---

# 1. 전체 그림부터 (한 눈에)

```text
1️⃣ 브라우저가 페이지 로드
2️⃣ 클라이언트에서 sw.js 등록 요청
3️⃣ 브라우저가 sw.js를 Service Worker로 기록
4️⃣ 이후 JS/CSS/HTML 요청 시
   → Service Worker가 가로챌 수 있음
```

---

# 2. 파일 구조 (Next.js App Router)

```text
app/
 ├─ layout.tsx
 ├─ page.tsx
 ├─ sw-register.tsx   ← 클라이언트 전용 등록 코드
public/
 └─ sw.js             ← Service Worker 스크립트 (필수 위치)
```

📌 **중요**

* `sw.js`는 반드시 `public/`에 있어야 한다
  → 그래야 `/sw.js`로 접근 가능
* Service Worker는 **빌드 결과물 기준으로 동작**

---

# 3. Service Worker 스크립트 (public/sw.js)

👉 **UI 캐싱만 하는 최소 예제**

```js
const CACHE_NAME = "app-shell-v1";
const ASSETS = ["/", "/favicon.ico"];

self.addEventListener("install", event => {
  event.waitUntil(
    caches.open(CACHE_NAME).then(cache => cache.addAll(ASSETS))
  );
});

self.addEventListener("activate", event => {
  event.waitUntil(
    caches.keys().then(keys =>
      Promise.all(
        keys
          .filter(key => key !== CACHE_NAME)
          .map(key => caches.delete(key))
      )
    )
  );
});

self.addEventListener("fetch", event => {
  const req = event.request;

  // API 요청은 관여하지 않음
  if (req.url.includes("/api")) return;

  event.respondWith(
    caches.match(req).then(res => res || fetch(req))
  );
});
```

📌 여기서 **중요한 포인트**

* React ❌
* JSX ❌
* Zustand ❌
* TanStack Query ❌
* **정적 파일 요청만 처리**

---

# 4. Service Worker 등록 코드 (클라이언트 전용)

Next.js App Router에서는 **무조건 Client Component**에서 등록해야 한다.

### app/sw-register.tsx

```tsx
"use client";

import { useEffect } from "react";

export function ServiceWorkerRegister() {
  useEffect(() => {
    if ("serviceWorker" in navigator) {
      navigator.serviceWorker
        .register("/sw.js")
        .then(() => {
          console.log("Service Worker registered");
        })
        .catch(err => {
          console.error("Service Worker registration failed", err);
        });
    }
  }, []);

  return null;
}
```

📌 왜 `use client`?

* Service Worker 등록은 **브라우저 API**
* 서버 컴포넌트에서는 접근 불가

---

# 5. layout.tsx에 한 번만 포함

```tsx
// app/layout.tsx
import { ServiceWorkerRegister } from "./sw-register";

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="ko">
      <body>
        <ServiceWorkerRegister />
        {children}
      </body>
    </html>
  );
}
```

📌 **중요**

* 한 번만 실행되면 됨
* 페이지 이동마다 재등록 ❌

---

# 6. 이 코드로 실제로 일어나는 일 (순서대로)

### 최초 접속 시

```text
1️⃣ layout.tsx 렌더링
2️⃣ ServiceWorkerRegister 실행
3️⃣ navigator.serviceWorker.register("/sw.js")
4️⃣ 브라우저:
   - "이 사이트는 Service Worker 있음" 기록
   - sw.js 다운로드
5️⃣ install 이벤트 실행
6️⃣ App Shell 캐시 저장
7️⃣ Service Worker 종료
```

---

### 이후 JS 번들 요청 시

```text
브라우저가 app.js 요청
→ Service Worker fetch 이벤트 실행
→ 캐시에 있으면 캐시 응답
→ 없으면 네트워크 요청
```

---

# 7. “이걸 왜 App Router에서 이렇게 하냐면”

* App Router는 기본이 **Server Component**
* `navigator`는 **브라우저 전용**
* 그래서:

    * Client Component + `useEffect`
    * layout.tsx에서 한 번만

이게 **가장 안전한 패턴**

---

# 8. 절대 하지 말아야 할 것

❌ Server Component에서 register
❌ page.tsx마다 register
❌ Zustand/TanStack Query를 sw.js에 넣기
❌ API 응답 캐싱

---

# 9. 문서에 그대로 써도 되는 정리 문장

```md
Next.js(App Router) 환경에서 Service Worker는
클라이언트 전용 컴포넌트에서 등록한다.
Service Worker 스크립트(sw.js)는 public 디렉토리에 위치하며,
브라우저는 해당 스크립트를 기반으로
정적 리소스 요청(fetch)에만 관여한다.
UI 렌더링과 서버 데이터 캐싱은
Service Worker의 책임 범위에 포함되지 않는다.
```

# 10. 한 줄 요약 (완전 핵심)

> **Next.js App Router에서 Service Worker 등록이란
> “클라이언트에서 브라우저에게
> `/sw.js`를 네트워크 중개자로 써도 된다고 알려주는 것”이다.**

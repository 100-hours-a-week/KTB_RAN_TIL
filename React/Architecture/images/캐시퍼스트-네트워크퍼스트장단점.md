아래는 **개발·운영 관점에서 꼭 필요한 수준으로 정리한**
**cache-first / network-first의 장단점 비교**다.
(개념 설명 과잉 없이, 설계 판단에 바로 쓰이는 포인트만)

---

## cache-first 전략

### 개요

* **캐시를 우선 사용**, 없을 때만 네트워크 요청
* 목적: **속도·오프라인 대응**

### 장점

* 초기 로딩 속도가 매우 빠름
* 네트워크 불안정/오프라인에서도 동작 가능
* 서버 요청 수 감소
* 정적 리소스에 대해 비용 대비 효과가 큼

### 단점

* 최신성 보장 어려움 (v1/v2 UI 불일치 가능)
* 캐시 무효화 전략이 없으면 오래된 리소스가 유지됨
* 배포 이후 변경 사항이 즉시 반영되지 않음
* 디버깅 시 “왜 수정했는데 안 바뀌지?” 문제가 발생하기 쉬움

### 적합한 대상

* 폰트, 아이콘, 로고
* 거의 변하지 않는 이미지
* 버전 고정된 정적 라이브러리

---

## network-first 전략

### 개요

* **네트워크를 우선 시도**, 실패 시 캐시 fallback
* 목적: **최신성 보장**

### 장점

* 항상 서버의 최신 상태를 우선 반영
* 배포 이후 UI/로직 불일치 문제 최소화
* 캐시로 인한 예측 불가능한 상태 감소
* 운영·디버깅이 상대적으로 단순

### 단점

* 네트워크 상태에 따라 초기 로딩이 느릴 수 있음
* 오프라인 환경에서는 캐시 없으면 동작 불가
* 서버 요청 수 증가 가능

### 적합한 대상

* index.html
* JavaScript 번들
* CSS
* 라우트 단위 chunk
* 최신 UI 반영이 중요한 화면

---

## 핵심 비교 요약

| 항목          | cache-first | network-first   |
| ----------- | ----------- | --------------- |
| 우선순위        | 캐시          | 네트워크            |
| 로딩 속도       | 매우 빠름       | 상대적으로 느릴 수 있음   |
| 최신성         | 낮음          | 높음              |
| 오프라인 대응     | 강함          | fallback 시에만 가능 |
| 캐시 무효화 난이도  | 높음          | 낮음              |
| UI 업데이트 안정성 | 낮음          | 높음              |

---

## 설계 판단 한 줄 요약

> **cache-first는 “변하지 않는 리소스”에,
> network-first는 “항상 최신이어야 하는 리소스”에 적용한다.**

이 기준만 지키면
두 전략을 **같이 써도 복잡해지지 않는다.**

## FSD 아키텍처란?
- 기능 단위로 분할

### 계층 구조
- 레이어
- 슬라이스
- 세그먼트

![FSD.jpg](images/FSD.jpg)

- 레이어
- 레이어의 순서는 의존성 방향입니다. app -> shared
  - **app** :  애플리케이션 실행에 필요한 핵심 요소가 포함됩니다 
    - 애플리케이션의 진입점, 전역 스타일, 라우팅, 프로바이더 등
    - 슬라이스가 없고, 세그먼트로 분할
  - **pages** : 전체 페이지 또는 중첩 라우팅의 핵심 영역을 담당합니다
  - **widget** : 독립적으로 동작하는 대형 UI 및 기능 블록입니다
  - **feature** : 전체 제품 기능의 재사용된 구현, 즉 사용자에게 비즈니스 가치를 제공하는 작업입니다.
  - **entity** : 프로젝트가 협력하는 비즈니스 엔티티(예: user또는 product).
  - **shared** : 프로젝트 전반에서 재사용되는 일반적인 유틸리티, UI 컴포넌트 등을 포함합니다
    - 슬라이스가 없고 세그먼트로 분할
    

- 슬라이스
  - 비즈니스 도메인 별
  - 같은 레이어 내의 다른 슬라이스 참조 불가 -> 높은 응집도와 갖은 결합도
  

- 세그먼트
  - 기술적 목적에 따라 
  - ui : UI 컴포넌트, 스타일 등 화면 표시와 관련된 코드
  - api :  백엔드 통신, 데이터 타입, 매퍼 등 **데이터 로직**
  - model : 스키마, 인터페이스, 스토어, 비즈니스 로직 등 **도메인 모델**
  - lib : 해당 슬라이스 내에서 공통으로 사용되는 라이브러리 코드
  - config : 환경 설정, 기능 플래그(Feature flags) 등


> [ FSD 관점에서 레이어 관계 ]

 ```    
    [ App ]          ← Composition Root (진입점)
    ↓
    [ Pages ]        ← 화면 단위 조합
    ↓
    [ Widgets ]
    ↓
    [ Features ]
    ↓
    [ Entities ]
    ↓
    [ Shared ]       ← 가장 범용적인 재료
   ```

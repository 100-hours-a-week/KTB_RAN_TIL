## 1. 코드 스플리팅(Code Splitting)

### 1.1 개념 (What)

코드 스플리팅은 **하나의 큰 JavaScript 번들을 여러 개의 작은 번들로 나누는 기법**이다.
→ 사용자가 **지금 필요한 코드만 먼저 로드**하고, 나머지는 나중에 로드한다.

---

### 1.2 왜 필요한가 (Why)

SPA / Next.js 앱에서 초기 JS 번들이 커지면 다음 문제가 발생한다.

* 초기 로딩 시간 증가 (TTI, FCP 악화)
* 불필요한 JS 파싱·실행 비용 증가
* 모바일 / 저사양 기기에서 체감 성능 저하

**핵심 목적**

* 초기 로딩 비용 최소화
* 페이지/기능 단위로 JS 로드 범위 제한

---

### 1.3 동작 방식 (How)

#### 기본 아이디어

```text
기존
└── main.js (모든 페이지 + 모든 기능)

코드 스플리팅 후
├── home.chunk.js
├── planner.chunk.js
├── chat.chunk.js
└── vendor.chunk.js
```

* 브라우저는 **현재 접근한 페이지에 필요한 chunk만 다운로드**
* 나머지는 접근 시점에 네트워크 요청

---

### 1.4 구현 방식

#### (1) Dynamic Import (표준 JS)

```ts
const ChatPage = () => import('./ChatPage');
```

#### (2) React.lazy

```tsx
const ChatPage = React.lazy(() => import('./ChatPage'));
```

#### (3) Next.js (App Router 기준)

```tsx
const ChatPage = dynamic(() => import('./ChatPage'), {
  ssr: false,
});
```

Next.js는 **페이지 단위 코드 스플리팅을 기본 제공**한다.
(`/app/chat/page.tsx` → chat 전용 번들 생성)

---

### 1.5 장단점

| 항목    | 장점          | 단점              |
| ----- | ----------- | --------------- |
| 초기 로딩 | JS 크기 감소    | 첫 진입 후 추가 요청 발생 |
| UX    | 빠른 첫 화면     | 코드 로딩 순간 지연 가능  |
| 네트워크  | 필요한 것만 다운로드 | 요청 횟수 증가        |

---

## 2. 프리패칭(Prefetching)

### 2.1 개념 (What)

프리패칭은 **“나중에 필요할 가능성이 높은 리소스를 미리 다운로드해두는 전략”**이다.

> 아직 사용하지 않지만
> **곧 쓸 것 같으면 백그라운드에서 미리 받아둔다**

---

### 2.2 왜 필요한가 (Why)

코드 스플리팅만 적용하면 다음 문제가 생긴다.

* 페이지 이동 시 **JS chunk를 새로 요청**
* 네트워크 지연 → 화면 전환이 느려 보임

프리패칭은 이를 보완한다.

---

### 2.3 동작 방식 (How)

```text
1. 사용자는 Home 페이지에 있음
2. 브라우저는 idle 상태
3. "다음에 갈 확률이 높은 페이지" JS를 미리 다운로드
4. 실제 이동 시 → 캐시에서 즉시 로드
```

---

### 2.4 Next.js에서의 프리패칭

#### (1) Link 컴포넌트 (기본 활성화)

```tsx
<Link href="/chat">채팅방</Link>
```

* viewport에 들어오면 자동 prefetch
* `rel="prefetch"` 기반
* **유휴 시간 + 낮은 우선순위**로 로드

#### (2) 비활성화

```tsx
<Link href="/chat" prefetch={false}>
```

---

### 2.5 프리패칭 대상

* JS chunk
* route 관련 데이터 (RSC payload, loader 결과 등)

---

### 2.6 장단점

| 항목    | 장점           | 단점          |
| ----- | ------------ | ----------- |
| UX    | 페이지 전환 매우 빠름 | 실제로 안 쓰면 낭비 |
| 체감 성능 | 거의 즉시 전환     | 네트워크 자원 사용  |
| 모바일   | 빠른 화면 이동     | 데이터 요금 부담   |

---

## 3. 코드 스플리팅 vs 프리패칭 비교

| 구분 | 코드 스플리팅     | 프리패칭           |
| -- | ----------- | -------------- |
| 목적 | 초기 로딩 최소화   | 이동 시 지연 제거     |
| 시점 | **요청 시 로드** | **미리 로드**      |
| 대상 | JS 번들 분리    | 분리된 번들 사전 다운로드 |
| 관계 | 필수          | 보완 전략          |

> 프리패칭은 **코드 스플리팅이 전제 조건**이다.

---

## 4. 프로젝트 관점 정리 (Next.js + FSD 기준)

### 권장 전략

1. **페이지 단위 코드 스플리팅**

    * App Router 기본 활용
    * feature/widget 단위 lazy 로딩은 신중히 적용

2. **프리패칭은 기본값 유지**

    * 메인 → 상세 이동 흐름은 체감 성능에 유리

3. **프리패칭 비활성화 고려 케이스**

    * 관리자 페이지
    * 사용 빈도 낮은 설정 화면
    * 네트워크 비용이 큰 페이지

---

## 한 줄 요약

* **코드 스플리팅**: 지금 필요한 코드만 로드하도록 **JS를 쪼개는 전략**
* **프리패칭**: 나중에 쓸 가능성이 높은 코드를 **미리 받아두는 전략**

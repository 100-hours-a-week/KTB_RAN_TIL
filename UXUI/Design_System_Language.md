

## **디자인 시스템이란?**

서비스 목적에 맞게 끔 **일관되게 구성된 일련의 패턴(Design)과 공유된 규칙언어(System)를** 의미한다.

- 디자인 시스템을 일종의 언어로 이해한다.
- 그냥 단순히 모여져있는 디자인 요소 만으로는 시스템이 아니라 언어처럼 **구조가 있어야 진짜 시스템이다.**

---

### 디자인 시스템의 언어

| 언어 요소 | 디자인 시스템 대응 |
| --- | --- |
| 알파벳 | 디자인의 최소 단위 (예: 색상, 타이포) |
| 문법 | 컴포넌트의 조합 규칙 (언제 어떤 버튼을 쓰는지) |
| 발음 | UX 표현 방식의 일관성 (표현을 어떻게 명확히 할지) |

**디자인 시스템은 디자이너만 이해해서는 안 되고, 모든 구성원이 공유·활용할 수 있어야 한다**

- 규칙은 문서화 되어야하며,
- 왜 그것이 정답인지 맥락이 존재
- 모든 사람이 같은 규칙으로 해석할 수 있어야한다.

---

### 알파벳 = Foundation (기초 재료)

- 알파벳 : 버튼을 구성하기 위한 가장 작은 단위
- **Color / Typography = Foundation**
    - 알파벳은 **그 자체로는 의미가 없음**
    - 하지만 모든 문장의 **재료**

디자인 시스템에서

- Color
- Typography
- Spacing
- Radius
- Shadow

→ 쓸수있는 재료일뿐

---

## 문법 = Rules (가장 중요한 포인트)

- I + Wants ❌ → 문법
- Anatomy / Type = Rules

### 문법이란?

- 어떤 조합이 **허용되는지**
- 어떤 조합이 **틀린지**
- **언제** 무엇을 써야 하는지

### 버튼 예시

문법이 없다면 →

- “아무 버튼이나 쓰면 되지 않나?”
- “비활성 버튼을 여기 써도 되나?”
- “Outline 버튼을 primary 액션으로 써도 되나?”

문법을 통해 →

- 어떤 상황에 어떤 버튼을 써야 하는지
- Primary / Secondary / Disabled는 언제 쓰는지
- 동시에 몇 개까지 써도 되는지

---

## 발음 = 실제 사용 맥락 (운영 단계)

- rrrrrr like 현지인 → 발음

디자인 시스템 →

- 실무에서 디자이너/개발자가 **자연스럽게 쓰는지**
- 새로운 사람이 와도 **같은 판단을 하는지**
- 문서가 없어도 **“이건 이 버튼”**이라고 감이 오는지

시스템이 **몸에 배어 있는 상태 → 사용자 경험**

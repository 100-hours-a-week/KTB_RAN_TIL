

## 한 줄 요약

> **Git Flow는 “지금 서비스 중인 코드(main)”와 “다음에 나갈 코드(develop)”를 분리하고,
> 그 사이에서 기능·배포·긴급수정을 목적별 브랜치로 관리하는 전략이다.**

---

## 브랜치별 역할을 사람 역할로 비유

### 1. `main` — **실제 고객에게 보여지는 코드**

* **역할**

    * 운영 서버에 **배포된 상태 그대로**의 코드
* **특징**

    * 항상 안정적이어야 한다
    * 직접 개발하지 않는다
* **비유**

    * “지금 매장에서 판매 중인 제품”

---

### 2. `develop` — **다음 출시 버전을 준비하는 작업대**

* **역할**

    * 다음 배포 버전에 들어갈 기능들을 **모아두는 곳**
* **특징**

    * 기능 개발의 기준점
    * 테스트·통합이 계속 이루어진다
* **비유**

    * “다음 시즌 제품을 만드는 공장 작업대”

---

### 3. `feature/*` — **개별 기능 개발 공간**

* **생성 위치**

    * `develop`에서 분기
* **역할**

    * 하나의 기능 단위 개발
* **종료**

    * 개발 완료 → `develop`으로 merge
* **특징**

    * 여러 개가 동시에 존재 가능
* **비유**

    * “각 기능을 담당하는 개인 작업실”

---

### 4. `release/*` — **배포 직전 최종 점검 공간**

* **생성 위치**

    * `develop`에서 분기
* **역할**

    * 배포 직전 단계에서

        * 버전 번호 수정
        * 문구 수정
        * 치명적이지 않은 버그 수정
* **종료**

    * 배포 준비 완료 →

        * `main`에 merge (배포)
        * `develop`에도 merge (동기화)
* **비유**

    * “출시 전 QA·포장·라벨 붙이는 단계”

---

### 5. `hotfix/*` — **운영 중 긴급 수리 공간**

* **생성 위치**

    * `main`에서 분기
* **역할**

    * 운영 환경에서 발견된 **치명적인 버그** 즉시 수정
* **종료**

    * 수정 완료 →

        * `main`에 merge (즉시 배포)
        * `develop`에도 merge (다음 버전에 반영)
* **비유**

    * “판매 중인 제품의 리콜/긴급 수리”

---

## 전체 흐름을 한 번에 보기

```
feature/*  →  develop  →  release/*  →  main
                   ↑               ↓
                hotfix/*  ←─────────
```

---

## 왜 이렇게 나누는가? (핵심 이유)

### 1. **운영 안정성 확보**

* `main`에는 **검증된 코드만 존재**
* 개발 중 코드가 실수로 배포되는 것을 구조적으로 차단

### 2. **동시 작업 가능**

* 여러 `feature`를 동시에 개발 가능
* 배포 준비 중에도 긴급 수정(`hotfix`) 가능

### 3. **배포 이력 명확**

* 어떤 기능 묶음이 어떤 버전으로 나갔는지 추적 가능
* 릴리즈 단위 관리가 쉬움

---

## 실무에서 꼭 기억할 핵심 규칙

* `main`, `develop`에는 **직접 커밋하지 않는다**
* 모든 변경은 목적 있는 브랜치를 통해서만 들어온다
* `hotfix`와 `release`는 **양쪽(main, develop) 모두에 merge**한다

---

## 마지막 정리 

* **main**: 지금 서비스 중
* **develop**: 다음 서비스 준비 중
* **feature**: 기능 하나 만들기
* **release**: 배포 직전 정리
* **hotfix**: 급한 불 끄기

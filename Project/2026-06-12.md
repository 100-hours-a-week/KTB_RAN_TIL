# react가 실행되는 동작 원리

main.tsx가 시작점이다.
html확장자 코드에서 dom을 만드는데, vanillaJS든, react든 둘다 결과물은 html로 만든 dom이다.

```html
<!DOCTYPE html>
<html lang="ko">
  <head>
    <meta charset="UTF-8" />
    <link rel="icon" type="image/svg+xml" href="/favicon.svg" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <meta
      name="description"
      content="가상 스크롤 렌더링 방식을 직접 구현하고 비교하는 실험실"
    />
    <title>Scroll Lab</title>
  </head>
  <body>
    <div id="root"></div>
    <script type="module" src="/src/main.tsx"></script>
  </body>
</html>
```

해당코드에서 첫번째로

```html
<body>
  <div id="root"></div>
  <!--1-->
  <script type="module" src="/src/main.tsx"></script>
  <!--2-->
</body>
```

을 찾아볼수있는데,1번은 react 애플리케이션에 들어갈 빈 html 요소를 말한다.
main.tsx에서 root를 찾습니다. : geTelementById("root") 코드라인으로 인해 dom에서 root를 찾는다.

이후 vite가 tsx를 브라우저에서 실행될수있는 javascript로 변환한다.
-> 여기서 생각해볼수있는 점은 tsx가 결국 빌딩 후 브라우저에서 실행될수있는 js로 다시 변환된다면, ts는 개발자의 편의성을 위해 나온 언어인가? 라는 점을 생각해볼수있다.

이후 js로 변경된 main.tsx가 실행된다.

```tsx
//main.tsx
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import { App } from "./App";
import "./index.css";

createRoot(document.getElementById("root")!).render(
  <StrictMode>
    <App />
  </StrictMode>
);
```

document.getElementById('root') : index.html의 이 요소를 찾겠다. -> 마침 html 코드에 <div id="root"></div> 가 있다. 이로 인해 dom에는 root를 id로 갖는 div 공간이 존재.
createRoot(...) : 찾은 div 공간을 react가 관리할 렌더링 시작점을 등록한다. 이후 render 함수를 호출하여 app의 컴포넌트가 div안에 들어간다.

### ❓왜 body 전체가 아니라 root라는 요소를 사용하는 걸까?

react가 관리하는 영역과 일반 html영역을 구분하기 위해서이다.
사진과 같이 index.html에 있는 공간은 위에 있는 코드 처럼 root공간을 말하며, main.tsx에 있는 코드를 실행했을 때 나오는 코드가 실행된다.
![alt text](<src/Group 1.png>)

### ❓createRoot코드가 실행되면 어떤 일이 일어나나요?

createRoot는 리액트함수이다. FiberRoot와 HostRoot Fiber가 생성되고, root.render()함수가 실행되면서 App 컴포넌트가 렌더링이 된다.
HostRoot Fiber의 업데이트 큐에 렌더링이 업데이트가 되고, react reconciler 재조정 단계를 지납니다. 리콘실러재조정 단계 : Fiber 트리를 순회하며 변경부분이 계산됩니다. 이후 react dom 렌더가 되고 실제 dom에 반영된다.
즉, createRoot 코드에서 실행되면서, react 루트 구조가 생성되고, root.render가 실행되어 렌더링을 예약합니다. 이후 렌더 파즈 단계 jsx는 실행하면 객체가 생성된다.

```jsx
<App />
```

```jsx
{
  type: App,
  props: {}
}
```

이렇게 만들어진 객체를 이전 가상dom과 이번에 새로 만들어진 dom과 비교하는 reconciliation 단계가 존재한다. 재조정이라는 뜻은 말 그대로 기존에 그렸던 트리를 다시 조정하겠다 라는 말로, 재조정인 것이다.

1. render phase : reconciliation
2. commit phase : 재조정단계에서 계산된 변경을 realDOM에 반영한다.

### ❓index.html이 없어요.

vite 기반 React SPA에서는 index.html이 애플리케이션의 진입점이다.
브라우저가 index.html을 받은 후 main.tsx를 js코드로 변환해 실행하고, #root요소에 렌더링한다. 즉, 서버는 html 코드를 실행하지 않고 전송하는 역할을 한다.
반면 next app router에서는 개발자가 index.html나 main.tsx를 관리하지 않는다. nextJS에서는 url에 대응하는 page.tsx와 layout.tsx를 찾고, 서버 컴포넌트를 서버에서 직접 실행하여 html과 rsc payload를 생성한 뒤 결과물을 브라우저에 전달한다.

일단 여기까지 궁금한 부분은 나중에 더 딥다이브해야겠다. 오늘의 목표는 스크롤코드 직접 구상이다. 필요 시 바닐라JS 코드를 직접 확인하면서 공부해보겠다.

# SPA의 동작원리

## url

원래 html에서 url이 변경되면, 네트워크를 통해 새로운 html을 요청한다. 하지만 SPA는 처음 브라우저 접속시 서버로부터 단한번만 컴포넌트를 받는다.

가능한 이유는, url을 history api로 주소를 가져와서 split로 필요한 부분만 끊은 후 url 파라미터로 부터 받는다

routerPage의 역할은 해당 url 파라미터가 요청되면, 어떤 컴포넌트를 실행하면 좋을지 매핑되어있다.

```js
export const routerPage = {
  "/login": { page: LoginPageVDOM, effect: loginEffects, state: LoginState },
  "/signup": {
    page: SignupPageVDOM,
    effect: signupEffects,
    state: signupState,
  },
  "/home": {
    page: BoardHomePageVDOM,
    effect: [BoardHomeEffect, imgButtonEffect],
    state: BoardHomeState,
  },
  // "/post/get": {
  //   page: () => h("div", {}, "loading..."),
  //   effect: null,
  //   state: {},
  // },
  // "/user/info": handleUserInfo,
  // "/user/password-modify": handlePasswordModify,
  // "/post/update": BoardPostUpdatePage,
};
```

이건 내가 직접 작성했던 vanillaJS 코드인데, 이때 /login이라는 파라미터를 받고 각각의 컴포넌트를 부여 받는다.

```
const pathname = window.location.pathname
const matched = routerPage[pathname] //만들어둔 routerPage라는 객체와 연결
matched.page() //컴포넌트 렌더
```

### 동작 흐름

1. 링크 클릭 -> pushState로 URL 변경 (서버에 요청하지 않는다)
2. window.location.pathname으로 url을 string으로 가져온다 ("/home")
3. routerPage에서 매칭되는 라우팅 찾기
4. 매핑된 page만 새로 렌더한다.

이건 vanillaJS로 작성했을때는 이렇게 설계하지만, jsx일 경우에는 어떻게 작성할까?

## jsx일 경우 router

```jsx
// Vanilla의 routerPage 객체 → React Router의 Routes
import { Routes, Route } from "react-router-dom";

function App() {
  return (
    <Routes>
      <Route path="/login" element={<LoginPage />} />
      <Route path="/signup" element={<SignupPage />} />
      <Route path="/home" element={<BoardHomePage />} />
    </Routes>
  );
}
```

### Vanilla에서 직접 했던 게 어디로 갔나

- routerPage 객체 매핑 -> Routes, Route
- window.pathname읽기 -> react router가 내부에서 자동으로 계산
- page() 렌더링 -> element={<>}
- effect -> 컴포넌트 안 useEffect
- state -> 컴포넌트 안 useState

mogi_logi에서 연결했던 page/state/effect를 jsx로 마이그레이션

```jsx
function BoardHomePage() {
  const [state, setState] = useState(BoardHomeState); // ← state

  useEffect(() => {
    // BoardHomeEffect, imgButtonEffect 로직     ← effect
  }, []);

  return <div>...</div>; // ← page (VDOM)
}
```

## context api

그럼 context는 provider로 감싼 범위 안에서 전역객체처럼 사용되는, props로 전달하지 않아도, useContext 훅을 통해 언제든 정보를 꺼낼수있다.

```jsx
// Vanilla 비유
const RouterContext = {
  currentPath: "/virtual-scroll/fixed",
  params: { mode: "fixed" },
  navigate: (to) => { history.pushState(...); renderRoute(); }
};
```

routerProvider가 이 routerContext를 전역에 "문맥"을 뿌려주는 것이다. 이와 관련된 훅(예 : useParams())로 정보를 반환할 수 있다.

이부분을 실제로 까먹어서 다시 복습했다. 기술블로그에 작성해두기.

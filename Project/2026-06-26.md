# 코딩테스트

### 익명함수

- 자바스크립트에선 일급함수, 익명함수를 () ⇒ {} 이렇게 나타냈던 것 처럼
- **파이썬**에선, **lambda** 를 사용한다.
  ```python
  JS:      (a, b) => a + b
  Python:  lambda a, b: a + b
  ```
  **이렇게도 가능**
  ```python
  f = lambda x: x * 2
  print(f(3)) # 6
  ```

### 정렬

- **객체 정렬**일 경우 key를 이용해서 정렬한다.
- **객체 정렬은 lambda 함수의 반환값을 기준으로 정렬한다**
  ```python
  sort(key=lambda x : x.kor) #오름차순
  sort(key=lambda x : -x.kor) #내림차순
  ```

### Tuple

- 튜플 특징
  - 두개의 튜플(묶음)을 비교하려면, (a,b) (c,d) 이 두개를 비교하려고 하면 앞에 있는 첫번째부터 비교하고 이후에 두번째를 비교한다
    - 예시
      (2,3) > (2,1) ⇒ False
      첫번째 순서 통과시 두번째를 비교한다.
  - 이것과 람다의 반환을 이용하여 lambda x : (x.kor, x.eng) 이렇게 하면 뒤에 반환하는 두가지 필드를 기준으로 비교한다.
  - 이것이 튜플 3개, 튜플 4개묶음이어도 같다.
  - tuple은 **인자**를 하나만 받는다.
    - 그래서 tuple안에 인자를 다른곳에서 가져오고 싶다면 붙이는 방법이 있다.
      ```python
      (i,)+tuple(map(...))
      #실제 코드
      points = [(i,)+tuple(map(int, input().split())) for i in range(1,n+1)]
      ```
    ```python
    tuple([1, 2, 3])           # (1, 2, 3)  ← 리스트 하나를 튜플로
    tuple(map(int, ...))       # map 하나를 튜플로
    tuple(i, map(...))         # ❌ 두 개는 안 됨
    ```

### cmp_to_key()

- 기준이 복잡할때 기준을 정해주는 함수를 사용할 수 있는 **cmp_to_key(compare)**
  - 1 우리가 원하는 순서
  - -1 반대

```python
from functools import cmp_to_key

students = [
    Student(90, 80, 90), # 첫 번째 학생
    Student(20, 80, 80), # 두 번째 학생
    Student(90, 30, 60), # 세 번째 학생
    Student(60, 10, 50), # 네 번째 학생
    Student(80, 20, 10), # 다섯 번째 학생
]

# custom comparator를 직접 정의
# x가 앞에 있는 학생, y가 뒤에 있는 학생이라 가정했을 때
# 이 순서가 우리가 원하는 순서라면 0보다 작은 값을,
# 반대라면 0보다 큰 값을
# 둘의 우선순위가 동일하다면 0을 반환하면 됩니다.
# 보통 반환값에 1, -1, 0을 사용합니다.
def compare(x, y):
    # x만 30의 배수라면 x가 더 앞에 있어야 하므로
    # 현재 순서가 옳습니다.
    if x.kor % 30 == 0 and y.kor % 30 != 0:
        return -1
    # y만 30의 배수라면 y가 더 앞에 있어야 하므로
    # 현재 순서는 틀렸습니다.
    if x.kor % 30 != 0 and y.kor % 30 == 0:
        return 1
    # 우선 순위가 동일한 경우입니다.
    return 0

# compare 기준에 따른 custom 정렬
students.sort(key=cmp_to_key(compare))

for student in students: # 정렬 이후의 결과 출력
    print(student.kor, student.eng, student.math)

>> 90 80 90
   90 30 60
   60 10 50
   20 80 80
   80 20 10

```

### 어려워서 해설 봤는데 이해하는데에도 시간 걸렷던 문제

**정렬된 수 위치 알아내기**

- 양의 정수를 원소로 갖는 길이가 *N*인 수열이 입력으로 주어졌을 때, 이 수열을 오름차순으로 정렬했을 때 각각의 위치에 있던 원소가 어느 위치로 이동하는지 출력하는 코드를 작성해보세요.

```python
# 클래스 선언
class Number:
    def __init__(self, number, index):
        self.number, self.index = number, index

# 변수 선언 및 입력
n = int(input())
numbers = []
arr = list(map(int, input().split()))
numbers = [
    Number(num, i)
    for i, num in enumerate(arr)
]
answer = [
    0 for _ in range(n)
]

# Custom Comparator를 활용한 정렬
numbers.sort(key=lambda x: (x.number, x.index))

# 정렬된 숫자들의 원래 인덱스를 활용한 정답 배열 저장
for i, number in enumerate(numbers):
    answer[number.index] = i + 1 # 인덱스 보정

# 출력
for i in range(n):
    print(answer[i], end = ' ')

```

**여기서 킥은**

가장 처리가 힘들었던 부분이 같은 숫자를 어떻게 1번인지 2번인지 해결하는 것이었는데, 기존에 인덱스를 기억하고 있다가, 다시 돌려보낼때를 **answer[numer.originIndex] = sortedIndex+1** 로 처리햇더라.

```python
# Custom Comparator를 활용한 정렬
numbers.sort(key=lambda x: (x.number, x.index))

# 정렬된 숫자들의 원래 인덱스를 활용한 정답 배열 저장
for i, number in enumerate(numbers):
    answer[number.index] = i + 1 # 인덱스 보정
```

## 가변 높이 가상 스크롤 동작과정 딥다이브 중

## estimatedItemSize는 어디 있나

web.dev 공식 예제에는 `itemSize`만 있고 `estimatedItemSize`가 없어서 "이 prop이 실재하나?" 의심했는데, 확인 결과 **실재하는 optional prop**이 맞다.

- **공식 타입 정의**: `VariableSizeListProps`에 `estimatedItemSize`와 `itemSize`가 포함돼 있다. 정식 prop.
- **기본값 50**: react-window 소스에 `DEFAULT_ESTIMATED_ITEM_SIZE = 50`으로 정의돼 있다. 안 넘기면 50px로 추정.
- **게으른 계산 근거**: 소스의 `getEstimatedTotalHeight`가 측정된 행들의 크기 합 + 추정치를 조합해 전체 높이를 계산한다. `lastMeasuredIndex`로 "어디까지 측정했는지" 추적. → 우리가 정리한 게으른 계산이 코드로 확인됨.

# react-window 2.x에서는 API가 크게 변경됨

소스를 확인하다 보면 중요한 사실을 하나 더 알 수 있다.

우리가 흔히 블로그나 예제에서 보는 `FixedSizeList`, `VariableSizeList` 중심의 구조는 react-window 1.x 기준이다.

react-window 2.x에서는 API가 크게 변경되었기 때문에, 1.x 기준으로 정리한 내용을 그대로 적용하면 혼란이 생길 수 있다.

## 1. 컴포넌트 이름 변경

react-window 1.x와 2.x의 가장 큰 차이는 컴포넌트 이름이다.

| 1.x                                    | 2.x                      |
| -------------------------------------- | ------------------------ |
| `FixedSizeList`                        | `SimpleList` 또는 `List` |
| `VariableSizeList` / `DynamicSizeList` | `List`                   |
| `VariableSizeGrid`                     | `Grid`                   |

즉, 1.x에서 사용하던 `FixedSizeList`는 2.x에서 더 이상 같은 방식으로 사용되지 않는다.

실제로 이 변화 때문에 “react-window에서 `FixedSizeList`를 export하지 않는다”는 문제가 발생하기도 했다.

2.x에서 기존 컴포넌트명이 변경되었고, 1.8.x와 props 구조도 달라졌기 때문이다.

## 2. 동적 높이 처리 방식 개선

1.x의 `VariableSizeList`는 개발자가 item의 높이를 직접 알려주는 방식에 가깝다.

즉, react-window 1.x는 다음 전제를 가진다.

> “item의 높이는 개발자가 알고 있고, react-window는 그 값을 믿고 계산한다.”

그래서 높이가 나중에 바뀌면 개발자가 직접 `resetAfterIndex`를 호출해서 캐시를 무효화해야 했다.

하지만 2.x에서는 동적 리스트의 방향이 달라졌다.

동적 높이 처리를 위해 `ResizeObserver` 기반 측정 방식을 사용한다.

즉, 2.x에서는 개발자가 모든 높이를 미리 계산해서 넘기는 방식보다, 브라우저의 `ResizeObserver`를 통해 실제 렌더링된 요소의 크기를 측정하는 방식에 가까워졌다.

이 변화는 꽤 크다.

1.x에서 우리가 정리한 핵심은 다음과 같았다.

> react-window는 item 높이를 직접 측정하지 않는다.
>
> 개발자가 높이를 알려줘야 하고, 높이가 바뀌면 `resetAfterIndex`를 수동으로 호출해야 한다.

하지만 이 설명은 1.x 기준이다.

2.x에서는 `ResizeObserver` 기반의 자동 측정 흐름이 들어오면서, 동적 높이 처리 방식이 달라졌다.

즉, 우리가 별도로 만들던 커스텀 측정 훅이나 네이버 Big Table 사례에서 직접 구현했던 측정 로직을 react-window 2.x가 일부 기본 제공하는 방향으로 바뀐 것이다.

## 3. API 형태 변경

1.x에서는 다음처럼 children에 row 컴포넌트를 넘기는 방식이 일반적이었다.

```
<FixedSizeList
  height={500}
  itemCount={1000}
  itemSize={35}
  width={300}
>
  {Row}
</FixedSizeList>
```

하지만 2.x에서는 `rowComponent`, `rowCount`, `rowHeight`, `rowProps`처럼 더 명시적인 prop을 사용하는 형태로 바뀌었다.

주요 변경점은 다음과 같다.

| 1.x                      | 2.x                 |
| ------------------------ | ------------------- |
| `itemCount`              | `rowCount`          |
| `itemSize`               | `rowHeight`         |
| `itemData` / `data`      | `rowProps`          |
| children에 컴포넌트 전달 | `rowComponent` 전달 |

따라서 1.x 예제를 보고 2.x 프로젝트에 그대로 적용하면 prop 이름이나 컴포넌트 import 단계에서 문제가 생길 수 있다.

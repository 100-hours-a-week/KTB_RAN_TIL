### 1. ScrollItem[”category”][]

전 TIL에도 언급되었던 인덱스 접근 타입에 대한 응용이다.

```jsx
export interface ScrollItem {
  id: number;
  category: "Browser" | "React" | "Performance";
  // ...
}

const categories: ScrollItem["category"][] = [
  "Browser",
  "React",
  "Performance",
];
```

2차원배열을 뜻하는 것이 아니라

```jsx
**읽는 방법 : ScrollItem["category"] + []**
```

1. **ScrollItem["category"]<프로퍼티의 타입> :** 키로 프로퍼티 타입 꺼내기

   객체 타입에서 **특정 프로퍼티의 타입**을 꺼낸다.

   ```jsx
   ScrollItem["id"]; // number
   ScrollItem["title"]; // string
   ScrollItem["category"]; // **"Browser" | "React" | "Performance"**
   ```

2. **[]<배열>** : 그 타입의 1차원 배열

   ```jsx
   ScrollItem["category"][]
   ```

   - ("Browser" | "React" | "Performance")[]
   - 세 값 중 하나가 들어가는 평범한 1차원 배열

### 2. 배열 타입 []

```jsx
const experiments: { mode: ExperimentMode, step: string, label: string }[] = [
  { mode: "naive", step: "01", label: "Naive" },
  { mode: "fixed", step: "02", label: "Fixed height" },
  { mode: "dynamic", step: "03", label: "Dynamic height" },
];
```

```jsx
**읽는 방법 : { mode: ...; step: ...; label: ... }[]**
```

- 안쪽 {} : 원소 하나의 모양 - props
- 바깥 [] : 그런 객체가 여러개 모인 배열
  - **만약, const experiments: { mode: ExperimentMode; step: string; label: string } = []이라면?**
    - expoeriments는 props 모양이 { mode: ExperimentMode; step: string; label: string } 인 객체 하나인데, 뒤에 선언된 객체는 여러개이다.
- **인사이트 : 보통 맨처음 객체 및 반환값으로 알려주는 것이 : 타입 이다. 이것을 잘 이용하자**

[] 는 어떤 타입 뒤에든 붙는 **~의 배열** 이라는 표시

```jsx
string[]                  // 문자열들의 배열
ExperimentMode[]          // "naive"|"fixed"|"dynamic" 들의 배열
{ mode: string }[]        // 객체들의 배열
ScrollItem["category"][]  // (1번) 유니온들의 배열
```

객체 모양이 길면 type으로 이름 붙여준다.

type을 활용한 경우

```jsx
type Experiment = { mode: ExperimentMode; step: string; label: string }
const experiments: Experiment[] = [ ... ]
```

### 3. NavLink render props

- isActive 이런건 어디서 오는걸까요?

```jsx
<NavLink
  to="/fixed"
  className={({ isActive }) => (isActive ? "tab is-active" : "tab")}
/>
```

- { isActive } vs isActive의 차이 : 구조 분해할당이라는 뜻.

  - { flour, sugar, salt } = bread 처럼 bread객체 안에있는 프로퍼티를 구조 분해할당으로 받겠다는 뜻이다.
  - **그중 isActive를 꺼낸다.**
    - **그럼 props로 쓰면 되면 괜찮지 않을까? 괜찮다.**
    - props객체 안에 있는 isActive를 꺼낸다는 뜻이기 때문에 props.isActive를 하면괜찮다.

  ```jsx
  // ✅ 객체에서 isActive만 꺼냄 (구조 분해)
  ({ isActive }) => isActive ? ...     // isActive = true/false

  // ⚠️ 객체 전체를 isActive라고 이름 붙임 → 항상 truthy → 버그!
  (isActive) => isActive ? ...         // isActive = { isActive, ... } 객체

  // 방식 2를 제대로 쓰려면
  (props) => props.isActive ? ...
  ```

- **isActive는 누가 주는 것인가?**
  - NavLink 안에 있는 기본 프로퍼티 3가지가 있다. ⇒ NavLinkRenderProps : 함수가 받는 객체
    - 출처: https://api.reactrouter.com/v7/types/react-router.NavLinkRenderProps.html
    ```jsx
    type NavLinkRenderProps = {
      isActive: boolean,
      isPending: boolean,
      isTransitioning: boolean,
    };
    ```
    | 프로퍼티        | 언제 true                                  | 조건                     |
    | --------------- | ------------------------------------------ | ------------------------ |
    | isActive        | **이 링크가 현재 페이지일 경우**           | 항상 사용 가능           |
    | isPending       | 이 링크로 이동중(도착 전, ㅣoader 로딩 등) | data 모드 한정           |
    | isTransitioning | view Transition 중                         | `viewTransition` 켜야 함 |
- isPending 조건에 있는 data모드는, **앞서 공부했던 createBrowserRouter + RouterProvider (Data Router)**를 써야 동작한다.
- 즉, BrowserRouter만 쓰면 isPending은 data를 라우트 url 이동시에 사용하지 않기 때문에 항상 **false**이다.
  - 실무에서는 isActive >>>> isPending > isTransitioning 이렇게 사용한다고 함.

### 4. generateItems : reverse-engineering

**리버스 엔지니어링 :** 완성된 소프트웨어를 해체하며 분석하여 그 작동 원리 및 설계도를 역으로 파악하는 기술

이런 기법이 따로 부르는 방법이 있는진 몰랐는데, 요즘 내가 코드를 작성하면서 한번에 코드를 바로 나올수없다는 것을 알게 된 후로 AI가 항상 완성된 코드를 주면, **그 코드가 실제로 작성하면 왜, 어떻게 코드가 작성되었는지 알기 위해 코드의 태초로 되돌려보는 작업을 진행한다.**

```jsx
export function generateItems(count: number): ScrollItem[] {
  return Array.from({ length: count }, (_, index) => {
    const id = index + 1;
    const detailCount = (index % detailFragments.length) + 1;
    return {
      id,
      title: `Rendering sample ${id.toString().padStart(5, "0")}`,
      summary: `${id.toLocaleString()}번째 아이템의 렌더링 상태를 추적합니다.`,
      details: detailFragments.slice(0, detailCount).join(" "),
      category: categories[index % categories.length],
    };
  });
}
```

위 코드의 진화 가설

1. 손으로 데이터 직접 박기 → 테스트 데이터 필요
2. for 루프로 생성 → 가상 스크롤은 수천개가 필요함 손으로 못씀
3. for + push = Array.from으로 교체 → 더 선언적
4. scrollItem[] : 리턴 객체가 ScrollItem 모양에서 어긋나면 컴파일 에러로 잡기

> **리버스 엔지니어링 인사이트 :** 이 줄이 뭘하는지를 넘어, <**왜 여기 있는지>**를 물으면 AI가 짠 코드도 의도가 드러난다. **면접에서 코드 설명 요청시 동작과 의도를 말하는 것을 목표로**

### 5. react-window vs TanStack Virtual

|        | react-window                | TanStack Virtual                              |
| ------ | --------------------------- | --------------------------------------------- |
| 방식   | 컴포넌트 제공               | headless 훅                                   |
| 마크업 | 정해진 구조에 맞춰야됨      | 직접 다 작성해야됨.(라이브러리는 계산만 해줌) |
| 자유도 | 낮음(구조가 정해져있음 )    | 높음                                          |
| 번들   | -2KB                        | 가상화 로직 -5KB                              |
| 강점   | 단순, 균일한 리스트, 경량화 | 동적 높이, 복잡한 구조, 양방향 스크롤         |

- react - window는 구조가 정해져있음.
- TanStack virtual은 위치만 계산해줌. → 구조 직접 작성
  - 최근 트렌드 : react-window는 고정 크기 한계로 지배력 약화 TanStack Table+Virtual 조합을 많이 쓴다고 함.
- 프로젝트 진행 :
  - fixed height : react-window와 비교
  - dynamic height : TanStack Virtual의 영역
- **구현 핵심**
  - 보이는 범위 계산 (start, end index)
  - overscan(버퍼) : 직접 구현시 빼먹기 쉬운 부분이라고 함 → 빠른 스크롤 시 blank 깜빡임 방지 → 라이브러리엔 거의 항상 있다.
  - dynamic height 위치 보정

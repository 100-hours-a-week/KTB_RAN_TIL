### **왜 style에 absolute와 top으로 하는걸까?**

가상 스크롤은 전체 1만 개 중 보이는 15개 정도만 DOM에 그립니다. 예를 들어 13번 아이템부터 그려야 한다면, 1~12번 아이템은 DOM에 존재하지 않습니다.

absolute는 상위 요소를 기준으로 dom에 절대좌표 (top)에 배치합니다. 덕분에 dom 순서와 화면상의 위치를 분리할 수 있습니다.

dom에선 첫 자식이지만 화면에서는 900xp지점 배치가 viewport 밖의 좌표라도 상관없이 둘 수 있습니다.

이 absolute의 이 특성을 윈도잉에서 활용합니다.

**마치 창문 처럼 렌더링 전체 캔버스에서 절대적인 top 위치에 아이템을 그립니다.**

어떤 아이템의 top의 위치가 100인데 scorlltop이 120으로 그 아이템은 화면 위쪽으로 20px 밀려나 viewport에서 벗어난 상태입니다.

이 아이템은 어차피 우리 눈에 보이지 않기 때문에 윈도잉 기법대로 굳이 그리지 않아도 됩니다.

**아이템 위치가 viewport 범위 안에 들어온 경우에만 재빠르게 그리면 됩니다.**

아래 영상은 마우스의 위치를 스크롤로 내렸을때 실제 가상 스크롤에서 absolute top방식이 영향을 끼치는 것을 눈으로 확인할 수 있는 영상입니다. 만약 저 격자를 전체 viewoport라고 했을때, 스크롤이 된다고 느껴지게끔 하는 것입니다.

#### **absolute 말고 다른 방법은 없을까요?**

top을 바꾸면 브라우저는 해당 요소의 위치를 레이아웃 단계에서 다시 계산합니다.

absolute 요소라 다른 형제 요소까지 밀리는 전체 리플로우를 일으키지 않지만, 레이아웃단계에 관여하게 됩니다.

레이아웃, 페인팅 단계를 최적화를 위해 transform : translateY(top)을 사용해볼수있습니다.

**transform은 레이아웃과 페인트 단계를 건너뛰고 마지막 합성(composite) 단계에서만 처리됩니다.**

또한 transform이 걸린 요소는 별도 레이어로 승격되기 쉽고, 그 레이어의 합성은 CPU가 아닌 GPU가 담당할 수 있어 성능에 유리합니다.

실제로 TanStack Virtual 공식 문서의 기본 예제도 보이는 아이템을

position: absolute로 두고 transform: translateY(virtualItem.start)로 위치를 잡습니다.

핵심은,

**absolute + top이든 absolute + transform이든 근본 동작 원리는 같다는 점입니다.**

둘 다 요소를 좌표로 배치하는 것이고, transform은 그 중 더 저렴한 단계에서 좌표를 적용하는 최적화일 뿐입니다.

# 프로젝트 : 동적 높이 가상 스크롤과 네이버 Big Table 사례

고정 높이에서는 모든 행의 높이가 같기 때문에 계산이 단순했다.

```tsx
offset = index * itemSize;
startIndex = Math.floor(scrollTop / itemSize);
totalHeight = itemCount * itemSize;
```

하지만 행마다 높이가 달라지면 이 공식이 모두 깨진다.

예를 들어 어떤 행은 60px, 어떤 행은 120px, 어떤 행은 200px이라면 `index * itemSize`로는 해당 행이 어디서 시작하는지 알 수 없다.

결국 동적 높이에서는 다음 세 가지 문제를 다시 풀어야 한다.

```
1. 각 행의 위치를 어떻게 계산할 것인가?
2. 현재 scrollTop에 해당하는 시작 index를 어떻게 찾을 것인가?
3. 아직 보지 않은 영역의 전체 높이는 어떻게 계산할 것인가?
```

이 문제를 해결하기 위해 가변 높이 가상 스크롤에서는 보통 **누적 오프셋 캐시**, **이진 탐색**, **게으른 계산과 추정치**를 사용한다.

---

## 1. 누적 오프셋 캐시

동적 높이에서는 각 행의 높이만 알아서는 부족하다.

중요한 것은 **각 행이 어디서 시작하는지**, 즉 offset이다.

```
height: [ 60, 120,  80, 200,  95, ...]
offset: [  0,  60, 180, 260, 460, ...]
```

0번 행은 0px에서 시작한다.

1번 행은 0번 행의 높이 60px만큼 내려온 60px에서 시작한다.

2번 행은 0번과 1번 행의 높이를 더한 180px에서 시작한다.

즉 offset은 “해당 행 이전까지의 높이를 모두 더한 값”이다.

```
offset[n] = 0번부터 n-1번까지의 높이 합
```

처음에는 “어차피 각 행의 높이를 아는데 왜 offset 배열이 필요하지?”라고 생각했다.

하지만 **높이를 아는 것과 위치를 아는 것은 다르다.**

행의 높이는 그 행 하나의 크기이고, offset은 그 행이 전체 리스트 안에서 시작하는 위치다.

가변 높이에서는 앞의 행들이 모두 다른 높이를 가지기 때문에, 위치를 구하려면 앞 행들의 높이를 누적해야 한다.

그래서 한 번 계산한 offset을 배열에 저장해두고, 나중에 다시 필요할 때는 다시 더하지 않고 꺼내 쓴다.

```tsx
const offsets = new Float64Array(itemCount);
const sizes = new Float64Array(itemCount);

let totalHeight = 0;

for (let index = 0; index < itemCount; index += 1) {
  offsets[index] = totalHeight;
  sizes[index] = measuredSizes.get(index) ?? estimatedItemHeight;
  totalHeight += sizes[index];
}
```

이 방식은 메모리를 조금 쓰는 대신 계산 속도를 얻는 방식이다.

예를 들어 `Float64Array`는 숫자 하나당 8byte를 사용한다.

```
10,000개 * 8byte = 약 80KB
offsets + sizes = 약 160KB
```

처음에는 “1만 개짜리 배열이면 메모리 낭비 아닌가?”라고 생각했지만, DOM 노드 1만 개를 실제로 렌더링하는 비용과 비교하면 훨씬 작다.

가상 스크롤은 무거운 DOM 노드를 줄이는 대신, 가벼운 숫자 배열을 들고 있는 방식이다.

즉, 이건 시간과 메모리의 트레이드오프이고 대부분의 경우 좋은 선택이다.

## 2. 이진 탐색

고정 높이에서는 현재 스크롤 위치가 몇 번째 행에 해당하는지 쉽게 구할 수 있었다.

```tsx
startIndex = Math.floor(scrollTop / itemSize);
```

하지만 가변 높이에서는 이 공식이 통하지 않는다.

행마다 높이가 다르기 때문에 `scrollTop / itemSize`를 할 수 없다.

대신 offset 배열에서 현재 scrollTop이 들어가는 구간을 찾아야 한다.

예를 들어 offset 배열이 다음과 같다고 하자.

```
offset: [0, 60, 180, 260, 460]
```

이때 `scrollTop = 300`이라면 300은 260과 460 사이에 있다.

따라서 현재 화면의 시작 행은 3번 행이다.

```
260 <= 300 < 460
```

여기서 중요한 점은 offset 배열이 항상 오름차순이라는 것이다.

높이는 음수가 될 수 없기 때문에, 누적값인 offset은 계속 증가한다.

정렬된 배열에서 특정 값이 들어갈 위치를 찾는 문제이므로 이진 탐색을 사용할 수 있다.

```tsx
function findItemIndex(
  offsets: Float64Array,
  sizes: Float64Array,
  targetOffset: number
) {
  let low = 0;
  let high = offsets.length - 1;

  while (low <= high) {
    const middle = Math.floor((low + high) / 2);
    const itemEnd = offsets[middle] + sizes[middle];

    if (itemEnd < targetOffset) {
      low = middle + 1;
    } else {
      high = middle - 1;
    }
  }

  return Math.min(low, offsets.length - 1);
}
```

이 함수는 `targetOffset`이 어느 행에 속하는지 찾는다.

각 행의 끝 위치는 다음처럼 계산한다.

```tsx
itemEnd = offsets[middle] + sizes[middle];
```

현재 행의 끝이 targetOffset보다 앞에 있으면 더 뒤쪽을 탐색하고, 아니면 앞쪽을 탐색한다.

덕분에 모든 행을 처음부터 훑는 O(n)이 아니라, O(log n)으로 시작 index를 찾을 수 있다.

1만 개의 아이템이 있어도 약 14번 정도의 비교로 찾을 수 있다.

## 3. 게으른 계산과 추정치

동적 높이에서는 모든 행의 실제 높이를 처음부터 알 수 없는 경우가 많다.

특히 아직 화면에 렌더링되지 않은 아이템은 실제 높이를 모를 수 있다.

그래서 가변 높이 가상 스크롤은 보통 처음부터 모든 것을 정확히 계산하지 않고, 방문한 영역은 실제 값으로 채우고 아직 방문하지 않은 영역은 추정치로 채운다.

```tsx
sizes[index] = measuredSizes.get(index) ?? estimatedItemHeight;
```

측정된 값이 있으면 실제 높이를 사용하고, 아직 측정되지 않았다면 `estimatedItemHeight`를 사용한다.

이 방식은 “일단 대략적인 높이로 전체 스크롤 영역을 만들고, 실제로 렌더링된 뒤 측정값이 생기면 보정하는 방식”이다.

react-window의 `VariableSizeList`에서는 이 추정치를 `estimatedItemSize`라는 prop으로 받는다.

```tsx
<VariableSizeList
  height={500}
  width={500}
  itemCount={items.length}
  itemSize={getItemSize}
  estimatedItemSize={100}
>
  {Row}
</VariableSizeList>
```

여기서 중요한 점은 `VariableSizeList`는 직접 높이를 측정하지 않는다는 것이다.

개발자가 `itemSize` 함수로 각 아이템의 높이를 알려줘야 한다.

즉, `VariableSizeList`는 다음 상황에 적합하다.

```
행마다 높이는 다르지만,
데이터만 보고 그 높이를 미리 계산할 수 있는 경우
```

예를 들면 채팅 메시지 길이에 따라 대략적인 높이를 계산하거나, 카드 타입에 따라 높이가 정해져 있는 경우다.

반대로 마크다운, 이미지 로딩, 반응형 줄바꿈처럼 실제로 그려봐야 높이를 알 수 있는 경우에는 직접 측정하는 방식이 더 적합하다.

## 4. 직접 구현한 동적 가상 스크롤 훅 이해

오늘 본 `useDynamicVirtualScroll` 훅은 react-window의 `VariableSizeList`를 그대로 쓰는 방식이 아니라, 측정 기반 가상 스크롤을 직접 구현한 형태였다.

핵심 상태는 두 가지다.

```tsx
const [measuredSizes, setMeasuredSizes] = useState(
  () => new Map<number, number>()
);

const [scrollTop, setScrollTop] = useFrameValue(0);
```

`measuredSizes`는 실제로 측정된 아이템의 높이를 저장한다.

```
Map<index, height>
```

처음에는 아무것도 측정하지 않았기 때문에 비어 있다.

행이 화면에 렌더링된 뒤 실제 높이를 측정하면 `measureItem(index, height)`를 통해 저장한다.

```tsx
const measureItem = useCallback((index: number, height: number) => {
  setMeasuredSizes((currentSizes) => {
    if (currentSizes.get(index) === height) {
      return currentSizes;
    }

    const nextSizes = new Map(currentSizes);
    nextSizes.set(index, height);

    return nextSizes;
  });
}, []);
```

여기서 좋은 점은 높이가 변하지 않았으면 기존 Map을 그대로 반환한다는 것이다.

```tsx
if (currentSizes.get(index) === height) {
  return currentSizes;
}
```

이렇게 하면 불필요한 리렌더링과 offset 재계산을 막을 수 있다.

이후 `metrics`에서 offset, size, totalHeight를 계산한다.

```tsx
const metrics = useMemo(() => {
  const offsets = new Float64Array(itemCount);
  const sizes = new Float64Array(itemCount);
  let totalHeight = 0;

  for (let index = 0; index < itemCount; index += 1) {
    offsets[index] = totalHeight;
    sizes[index] = measuredSizes.get(index) ?? estimatedItemHeight;
    totalHeight += sizes[index];
  }

  return { offsets, sizes, totalHeight };
}, [estimatedItemHeight, itemCount, measuredSizes]);
```

측정된 행은 실제 높이를 쓰고, 아직 측정되지 않은 행은 추정 높이를 쓴다.

그리고 각 행의 시작 위치를 offset 배열에 저장한다.

마지막으로 현재 스크롤 위치를 기준으로 보이는 범위를 찾는다.

```tsx
const visibleStart = findItemIndex(metrics.offsets, metrics.sizes, scrollTop);

const visibleEnd = findItemIndex(
  metrics.offsets,
  metrics.sizes,
  scrollTop + viewportHeight
);
```

그리고 overscan을 더해 실제 렌더링 범위를 넓힌다.

```tsx
const startIndex = Math.max(0, visibleStart - overscan);
const endIndex = Math.min(itemCount, visibleEnd + overscan + 1);
```

마지막으로 렌더링할 아이템 목록을 만든다.

```tsx
const virtualItems = Array.from(
  { length: endIndex - startIndex },
  (_, offset) => {
    const index = startIndex + offset;
    return {
      index,
      start: metrics.offsets[index],
    };
  }
);
```

이 결과로 컴포넌트는 전체 아이템을 모두 렌더링하지 않고, 현재 화면 근처의 아이템만 렌더링할 수 있다.

## 5. 네이버 D2 고성능 표 개선 사례로 본 가상 스크롤의 한계

오늘 함께 본 네이버 D2의 고성능 표 개선 사례는 지금까지 정리한 가상 스크롤 개념이 실제 서비스에서 어떤 문제로 이어지는지 보여주는 좋은 예시였다.

글의 구조는 크게 세 부분이었다.

```
1. 윈도잉의 보편 개념
2. React 가상화 라이브러리 비교
3. 네이버가 자체 개발한 Big Table 사례
```

### 5-1. 윈도잉은 프론트엔드만의 개념이 아니다

윈도잉은 프론트엔드 가상 스크롤에서만 쓰이는 개념이 아니다.

큰 데이터 전체를 한 번에 다루지 않고, 현재 필요한 일부 구간만 잘라서 처리하는 방식이다.

이 개념은 여러 분야에서 등장한다.

```
데이터 스트리밍
→ 텀블링 윈도, 슬라이딩 윈도, 세션 윈도

머신러닝
→ 시계열 데이터를 일정 길이의 시퀀스로 잘라 학습

컴퓨터 그래픽스
→ 뷰포트, 클리핑, 타일 기반 렌더링

프론트엔드
→ 현재 화면에 보이는 DOM만 렌더링하는 가상 스크롤
```

즉, 가상 스크롤은 윈도잉이라는 더 큰 개념을 프론트엔드 렌더링에 적용한 사례라고 볼 수 있다.

### 5-2. React 가상화 라이브러리 비교

대표적인 React 가상화 라이브러리에는 `react-virtualized`, `react-window`, `@tanstack/react-virtual`이 있다.

흐름은 대략 다음과 같다.

```
react-virtualized
→ 기능이 많지만 무겁다.

react-window
→ 더 작고 단순하다.
→ 리스트 가상화에 적합하다.
→ 다만 복잡한 테이블 요구사항에는 한계가 있다.

@tanstack/react-virtual
→ 비교적 최신이고 유연하다.
→ 직접 제어할 수 있는 범위가 넓다.
```

이 라이브러리들은 모두 공통적으로 비슷한 값을 필요로 한다.

```
viewport 높이
item 높이
item 개수
overscan 개수
```

결국 가상 스크롤의 기본 원리는 같다.

현재 스크롤 위치와 뷰포트 크기를 기준으로 “어떤 아이템을 렌더링할지” 계산하는 것이다.

---

### 5-3. 왜 react-window를 버렸을까?

네이버 사례에서 가장 흥미로웠던 부분은 `react-window` 기반 구현을 버리고 자체 표 컴포넌트인 Big Table을 만들었다는 점이다.

`react-window`는 리스트를 빠르게 렌더링하기 위해 각 아이템을 `position: absolute`로 배치한다.

이 방식은 일반 리스트에서는 매우 효과적이다.

하지만 표에서는 문제가 생긴다.

```
1. 실제 table 구조가 아니라 div 기반 구조가 된다.
2. td, colspan 같은 표 고유 기능을 쓰기 어렵다.
3. sticky header, 좌우 스크롤, 컬럼 리사이즈와 충돌한다.
4. 행 추가/삭제, 접기/펴기처럼 높이가 바뀌는 UI에서 좌표 계산이 복잡해진다.
```

즉, `absolute`는 리스트에서는 장점이지만, 복잡한 표에서는 약점이 될 수 있다.

이전에는 `position: absolute`가 DOM 순서와 화면 위치를 분리해 렌더링 성능을 높이는 방식이라고 이해했다.

그런데 표에서는 이 분리가 오히려 표 레이아웃의 자연스러운 동작을 깨뜨릴 수 있다.

특히 `<table>`, `<tr>`, `<td>`, `colspan`, `sticky header`처럼 브라우저가 원래 제공하는 표 레이아웃 기능과 잘 맞지 않는다.

### 5-4. resetAfterIndex가 실제 고통으로 등장한다

`react-window`의 `VariableSizeList`는 개발자가 알려준 높이를 믿는다.

즉, 라이브러리가 직접 높이를 측정하지 않는다.

그래서 아이템의 높이가 나중에 바뀌면 개발자가 직접 캐시를 무효화해야 한다.

```tsx
listRef.current.resetAfterIndex(index);
```

이 메서드는 “이 index 이후의 높이와 offset 계산을 다시 해라”라는 의미다.

처음에는 이게 단순한 API처럼 보였지만, 실제 서비스에서는 큰 부담이 될 수 있다.

예를 들어 로그 테이블에서 행을 접거나 펼치는 경우를 생각해보자.

```
접힌 상태
→ 높이 40px

펼친 상태
→ 높이 200px
```

이렇게 높이가 바뀌면 그 행 이후의 모든 offset이 달라진다.

그러면 개발자는 어느 index부터 다시 계산해야 하는지 알고, 직접 `resetAfterIndex`를 호출해야 한다.

네이버 Big Table 사례에서는 이런 수동 계산과 캐시 무효화 부담을 줄이기 위해 `ResizeObserver`를 내장해 높이를 자동 감지하는 방향을 선택했다.

즉, 이 사례는 다음 문장을 실제로 보여준다.

```
react-window는 측정하지 않고 개발자를 믿는다.
그리고 그 대가는 수동 캐시 무효화다.
```

---

### 5-5. absolute 방식의 비용

네이버 사례에서 특히 인상 깊었던 부분은 성능 수치였다.

`react-window` 기반 테이블은 컬럼 리사이즈 상황에서 프레임이 크게 떨어졌고, Big Table은 훨씬 안정적인 프레임을 유지했다.

여기서 원인으로 지목된 것이 `position: absolute` 기반 div 렌더링이었다.

리스트에서는 `absolute`로 각 행을 배치하는 방식이 효율적이다.

하지만 표에서는 셀 수가 많고, 컬럼 너비가 바뀔 때마다 수많은 셀의 위치와 크기를 다시 계산해야 한다.

특히 `top`, `left`, `width` 같은 레이아웃에 직접 관여하는 속성이 많이 바뀌면 브라우저의 레이아웃 계산 비용이 커질 수 있다.

즉, `absolute`는 무조건 좋은 방식이 아니라, 어떤 UI를 구현하느냐에 따라 장점과 단점이 갈린다.

```
단순 리스트
→ absolute 배치가 잘 맞는다.

복잡한 테이블
→ 실제 table 요소와 브라우저의 표 레이아웃 기능을 활용하는 편이 더 나을 수 있다.
```

이 부분에서 “가상 스크롤 구현 방식도 UI의 성격에 따라 달라져야 한다”는 점을 배웠다.

### 5-6. TanStack도 쓰지 않은 이유

처음에는 “그럼 react-window 대신 TanStack Virtual을 쓰면 되지 않나?”라고 생각할 수 있다.

하지만 네이버 사례에서는 TanStack 같은 오픈소스 라이브러리도 선택하지 않았다.

이유는 로그 시스템의 요구사항이 너무 특수했기 때문이다.

```
접기/펴기
컨텍스트 검색
라이브테일
스크롤 바닥 감지
향후 기획 변화 대응
```

이런 복잡한 도메인 요구사항을 오픈소스 라이브러리로 모두 맞추기 어렵다고 판단한 것이다.

즉, 측정 자동화만 필요했다면 TanStack이나 react-virtuoso 같은 선택지도 있었을 것이다.

하지만 요구사항이 로그 시스템에 너무 강하게 묶여 있었기 때문에 자체 개발을 선택한 것으로 보인다.

이 부분에서 “최신 라이브러리가 항상 정답은 아니다”라는 점을 배웠다.

라이브러리 선택은 기능 수나 최신성보다 **현재 도메인의 요구사항과 얼마나 잘 맞는지**가 더 중요하다.

---

### 5-7. 가상 스크롤도 OOM은 막지 못한다

마지막으로 가장 중요한 통찰은 이것이었다.

```
가상 스크롤은 렌더링 문제를 해결하지,
데이터 적재 문제를 해결하지는 않는다.
```

가상 스크롤을 사용하면 DOM 노드는 화면에 보이는 10~20개 정도로 유지할 수 있다.

하지만 무한 스크롤로 데이터를 계속 불러와 `items` 배열에 붙이면, JS 메모리에는 데이터가 계속 쌓인다.

```
DOM 노드
→ 10~20개만 유지됨
→ 가상 스크롤이 해결

items 배열
→ 10만 개, 30만 개까지 계속 쌓일 수 있음
→ 가상 스크롤이 해결하지 못함
```

예를 들어 로그 뷰어에서 사용자가 계속 아래로 스크롤하면 다음처럼 데이터가 누적될 수 있다.

```
처음: items = [0~99]
다음: items = [0~199]
다음: items = [0~299]
...
한참 뒤: items = [0~299999]
```

화면에는 여전히 20개 정도만 보이지만, JavaScript 메모리에는 30만 개의 데이터 객체가 남아 있다.

이 때문에 브라우저 메모리가 한계에 도달하면 OOM이 발생할 수 있다.

따라서 진짜 대용량 데이터를 다룰 때는 DOM만 줄이는 것으로는 부족하다.

데이터 자체도 관리해야 한다.

대안으로는 다음과 같은 방식이 있다.

```
1. 페이지 단위로 데이터 유지
2. 오래된 페이지는 메모리에서 제거
3. 필요하면 다시 fetch
4. IndexedDB에 캐시 후 JS 메모리에서는 비우기
5. React Query의 maxPages 같은 옵션으로 캐시 페이지 수 제한
```

즉, 가상 스크롤은 렌더링 레이어의 윈도잉이고, 대용량 데이터에서는 데이터 레이어에도 윈도잉이 필요하다.

## 오늘의 핵심 정리

동적 높이 가상 스크롤에서 중요한 것은 “높이가 다르다”는 사실보다, 그로 인해 기존 계산식이 깨진다는 점이다.

고정 높이에서는 위치, 시작 index, 전체 높이를 단순한 곱셈과 나눗셈으로 계산할 수 있었다.

하지만 가변 높이에서는 이 계산을 다시 설계해야 한다.

그래서 필요한 것이 다음 세 가지다.

```
1. 누적 오프셋 캐시
   → 각 행의 시작 위치를 저장해 위치 계산을 빠르게 한다.

2. 이진 탐색
   → scrollTop이 어느 행에 속하는지 빠르게 찾는다.

3. 게으른 계산과 추정치
   → 아직 모르는 높이는 추정값으로 두고, 측정되면 실제 값으로 보정한다.
```

그리고 이 세 가지는 `VariableSizeList`만의 특별한 기술이 아니라, 가변 높이를 다루는 가상 스크롤 구현이라면 공통으로 필요한 장치다.

차이는 높이를 어디서 얻느냐에 있다.

```
VariableSizeList
→ 개발자가 itemSize 함수로 높이를 미리 알려준다.

측정 기반 구현
→ 일단 렌더링한 뒤 실제 높이를 측정해서 저장한다.
```

네이버 Big Table 사례를 통해서는 가상 스크롤의 현실적인 한계도 확인했다.

```
1. absolute 기반 배치는 리스트에는 좋지만 복잡한 표에는 약점이 될 수 있다.
2. react-window는 높이를 직접 측정하지 않기 때문에 높이 변화 시 resetAfterIndex가 필요하다.
3. 복잡한 로그 테이블 같은 도메인에서는 오픈소스 라이브러리보다 자체 구현이 더 적합할 수 있다.
4. 가상 스크롤은 DOM 렌더링 수를 줄일 뿐, JS 메모리에 쌓인 데이터 자체를 줄이지는 못한다.
```

오늘 가장 크게 이해한 부분은 “높이를 아는 것”과 “위치를 아는 것”은 다르다는 점이다.

행의 높이를 알아도, 그 행이 어디서 시작하는지는 앞 행들의 높이를 누적해야만 알 수 있다.

그리고 한 가지 더 배운 것은, 가상 스크롤이 만능은 아니라는 점이다.

렌더링 문제는 해결할 수 있지만, 데이터가 계속 메모리에 쌓이는 문제는 별도로 해결해야 한다.

결국 대용량 UI에서는 두 가지 윈도잉이 모두 필요하다.

```
렌더링 윈도잉
→ 화면에 보이는 DOM만 렌더링한다.

데이터 윈도잉
→ 메모리에 필요한 데이터만 유지한다.
```

가상 스크롤을 이해할 때 DOM만 보는 것이 아니라, JS heap에 쌓이는 데이터까지 함께 봐야 한다는 점을 기억해야겠다.

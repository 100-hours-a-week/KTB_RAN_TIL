# TIL : TypeScript 타입 시스템 & React Router

> 한 줄 요약: **값(런타임)과 타입(컴파일 타임)은 다른 세계다.** 오늘 배운 거의 모든 게 이 구분에서 갈린다.

오늘 배운 키워드:

- `import type`
- `PropsWithChildren`
- `type` vs `interface`
- `as const`
- `typeof`(TS)
- 인덱스 접근 타입(`[number]`)
- 타입 단언(`as`) · 타입 가드(`mode is`)

## 1. `PropsWithChildren`

- props에 children을 합쳐주는 유틸 타입
- React가 제공하는 **타입 별칭**. 함수도 컴포넌트도 아니다. 정의는 이 한 줄이 전부:

```tsx
type PropsWithChildren<P = unknown> = P & { children?: ReactNode | undefined };
```

하는 일은 딱 하나 — **내 props에 `children?: ReactNode`를 교차 타입(`&`)으로 끼워 붙이는 것.**

```tsx
PropsWithChildren<{ title: string }>;
// 결국: { title: string; children?: ReactNode }
```

### 꼭 써야 하는 것은 아니다

직접 `children?: ReactNode`를 쓰는 것과 100% 동일하게 동작한다. 오히려 직접 선언을 선호하는 분위기이기도 하다.

- **props만 보면 인터페이스가 다 드러난다** (한 번 더 풀어볼 필요 없음)
- **children 타입을 좁히거나 필수로 만들 수 있다.** `PropsWithChildren`는 무조건 `children?: ReactNode`(옵셔널 + 거의 아무거나)로 고정이라, 아래 같은 제약을 못 건다.

```tsx
type Props = {
  children: ReactElement; // 필수 + 엘리먼트만 허용 → 직접 선언해야 가능
};
```

**쓰면 좋은 경우:** 제약 없는 단순 래퍼(레이아웃, 모달 컨테이너 등)에서 반복 줄이기 용도. 순전히 취향·팀 컨벤션.

## 2. `type` vs `interface`

`type`은 **타입에 이름을 붙이는** 키워드(타입 별칭). 런타임엔 사라지는 컴파일 타임 전용.

```tsx
type User = { name: string; age: number }; // 객체
type Size = "fixed" | "dynamic"; // 유니온
type Handler = (e: string) => void; // 함수 타입
type ID = string; // 원시 타입 별명
```

|                  | `interface`                       | `type`                                  |
| ---------------- | --------------------------------- | --------------------------------------- |
| 느낌             | **구조**를 정의 ("이런 모양이다") | **종류**를 정의 ("이런 것들 중 하나다") |
| 객체             | ✅                                | ✅                                      |
| 유니온           | ❌                                | ✅ (유니온은 `type`만 가능)             |
| 함수·원시 별명   | 제한적                            | ✅                                      |
| 같은 이름 재선언 | ✅ 자동 병합(declaration merging) | ❌ 중복 에러                            |

**기준:**

- 객체 모양 → `interface`(또는 `type`)
- 유니온/함수/별명 → `type`.
- **정리 : interface = 구조, type = 종류.**

## 3. `as const`

- 리터럴 + readonly + 깊게
- TS는 기본적으로 값을 **넓게(wide)** 추론한다.
  ```tsx
  const obj = { type: "fixed", count: 3 };
  // 추론: { type: string; count: number }   ← "fixed"가 아니라 string!
  ```
- `as const`를 붙이면 세 가지가 동시에 일어난다:
  1. **리터럴 타입으로 좁힘** (`string` → `"fixed"`)
  2. **전부 `readonly`** (배열은 readonly 튜플)
  3. **깊게(deep) 적용** — 중첩 객체·배열 안쪽까지
  ```tsx
  const obj = { type: "fixed", count: 3 } as const;
  // 추론: { readonly type: "fixed"; readonly count: 3 }
  ```

### 리터럴화는 "가능한 것"만

문자열·숫자·boolean만 리터럴이 된다. JSX 같은 건 리터럴화 안 되고 `readonly`만 붙는다.

```tsx
const meta = {
  naive: {
    title: "Naive rendering", // → readonly title: "Naive rendering" (리터럴)
    component: <NaiveList />, // → readonly component: JSX.Element (리터럴 아님)
  },
} as const;
```

## 4. `typeof`

- JS의 것과 TS의 것은 다르다.
- 이름만 같고 사는 동네가 다르다. **위치로 구분한다.**
  ```tsx
  // 값을 쓰는 자리 → JS typeof (런타임, 결과는 문자열)
  const t = typeof user; // "object"
  if (typeof x === "string") {
  }
  ```
  ```jsx
  // 타입을 쓰는 자리(`type ... =`, `:` 뒤) → TS typeof (컴파일 타임, 결과는 타입)
  type User = typeof user; // { name: string; age: number }
  ```

TS의 `typeof`는 "바꿔줘"가 아니라 **"이 값이 가진 타입을 가져와"**.

```tsx
const user = { name: "chaeyoung", age: 20 };
type User = typeof user;
// { name: string; age: number }  ← 각 필드 값을 보고 타입을 뽑아 조립 (중첩도 깊게)
```

값과 타입을 한 소스에서 관리 → 값만 고치면 타입이 자동으로 따라온다.

## 5. 인덱스 접근 타입 `[number]`

```tsx
export const EXPERIMENT_MODES = ["naive", "fixed", "dynamic"] as const;
export type ExperimentMode = (typeof EXPERIMENT_MODES)[number];
// "naive" | "fixed" | "dynamic"
```

- 두 줄의 정체가 완전히 다르다
  | | 1번째 줄 | 2번째 줄 |
  | --------- | -------------------------------------- | ---------------- | ------- | ---------- |
  | 만드는 것 | **값**(배열) | **타입**(유니온) |
  | 런타임에 | 살아있음 (순회·검증 가능) | 사라짐 |
  | 결과 | `readonly ["naive","fixed","dynamic"]` | `"naive"         | "fixed" | "dynamic"` |
- **`[number]` 은 무슨뜻인가**
  **숫자 인덱스로 접근 가능한 모든 요소의 타입**을 통째로 유니온으로 가져온다.
  ```tsx
  (typeof EXPERIMENT_MODES)[0](
    // "naive"  ← 0번 하나만
    typeof EXPERIMENT_MODES
  )[number]; // "naive" | "fixed" | "dynamic"  ← 전부
  ```

> 서랍 비유 :
> `[0]` = "0번 서랍 내용", `[number]` = "**숫자 붙은 서랍 전부의 내용**"

- **그냥 `type X = 'a' | 'b'`로 쓰면 안 되나? → 된다**
  타입만 보면 결과는 동일하다. 차이는 **런타임 배열이 있냐 없냐.**
  - 타입만 필요 → 직접 선언이 더 간단·명확
  - 목록을 런타임에도 써야 함(드롭다운 `.map`, `.includes` 검증, 순회) → `as const` + `typeof[number]` 패턴으로 **값과 타입을 한 소스에서 동기화**
  ```tsx
  EXPERIMENT_MODES.map((mode) => <option value={mode}>{mode}</option>); // 배열 있어야 가능
  EXPERIMENT_MODES.includes(input); // 배열 있어야 가능
  ```

## 6. 타입 단언 `as` vs 타입 표기 `:`

```tsx
EXPERIMENT_MODES.includes(mode as ExperimentMode);
```

`as :` **타입 단언(type assertion)**. → 이 값을 이 타입으로 봐줘

- **타입만 바꾼다** (정확히는 바꾼 척).
- **컴파일되면 사라진다.** 런타임 동작에 영향 0.

### 왜 여기서 `as`가 필요했나

`EXPERIMENT_MODES`(타입: `readonly ["naive","fixed","dynamic"]`)의 `.includes()`는 TS가 보기에 **`ExperimentMode`만 인자로 받겠다**고 돼 있다. 근데 넣는 `mode`는 `string | undefined`(더 넓음) → 타입 안 맞음 → 에러. 그래서 `as`로 "**그 타입인 척**" 달랜 것.

> **모순처럼 보이는 점:** "모드가 맞는지 확인하려고 includes 쓰는데, 인자엔 이미 모드라고 단언?" → 맞다, 논리적으론 어색하다. 하지만 `as`는 런타임에 사라지므로 실제 검사는 `mode` 진짜 값으로 정상 작동(`"banana"` → `false`). TS의 알려진 한계를 우회하는 관용 트릭.

### `:`와 `as`의 결정적 차이

|      | `:` (콜론)                     | `as`                                 |
| ---- | ------------------------------ | ------------------------------------ |
| 언제 | 변수·인자를 **처음 선언**할 때 | **이미 있는 값**의 타입을 바꿔 볼 때 |
| 뜻   | 이 변수는 이 타입이다          | 이 값을 이 타입으로 봐줘             |
| 예   | `const x: number = 3`          | `something as number`                |

```tsx
function isExperimentMode(mode: string | undefined) {
  // 선언 자리 → 콜론
  return EXPERIMENT_MODES.includes(mode as ExperimentMode); // 사용 자리 → as
}
```

**안 되는 시도들 (오늘의 실수 노트):**

```tsx
includes(mode : ExperimentMode)   // ❌ 문법 에러. : 는 선언 자리 전용
type mode = ExperimentMode;        // ❌ 헛수고. 'mode'라는 새 타입 별칭을 만들 뿐,
                                   //    이미 있는 인자 mode(값)의 타입은 안 바뀜
```

## 7. 타입 가드 `mode is ExperimentMode` (type predicate)

```tsx
function isExperimentMode(mode: string | undefined): mode is ExperimentMode {
  return EXPERIMENT_MODES.includes(mode as ExperimentMode);
}
```

함수의 **리턴 타입 자리**에 `boolean` 대신 쓴다.

하지만 이때 true 가 들어오면 값이 유효하면 mode는 `ExperimentMode` 이다. 라고 말해줌.

false가 들어오면 해당하지 않다 → ExperimentMode가 아님

- 안 붙여도 에러는 안 난다
  하지만, 차이는 호출하는 쪽에서 문제가 발생할수있었다.
  ```tsx
  if (!isExperimentMode(mode)) {
    return <Navigate to="/virtual-scroll/fixed" replace />;
  }
  // ★ 여기 아래:
  //   mode is 있음  → mode가 ExperimentMode로 좁혀짐
  //   : boolean     → mode가 여전히 string | undefined (또 as 필요)
  ```
  `true`/`false` 양쪽 다 좁혀준다:
  - `true` → `ExperimentMode`다
  - `false` → `ExperimentMode`가 아니다 (그래서 `!`로 쓰는 위 코드에서 동작)
    TS는 함수 속을 들여다보고 스스로 알아채지 못하므로, 우리가 직접 라벨을 붙여줘야 한다.

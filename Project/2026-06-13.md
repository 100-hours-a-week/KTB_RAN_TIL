# TIL: BrowserRouter는 왜 BrowserRouter고, RouterProvider는 왜 RouterProvider일까

> 라우터 설정 한 줄에서 출발해 JS 모듈 평가, JSX의 정체, React 렌더링 단계, 그리고 데이터 패칭 시점까지 한 줄기로 파고든 기록.

---

## Session 0. 왜 이걸 파고들게 됐나

페이지 라우팅 코드를 짤 때 나는 `BrowserRouter` 방식을 자주 썼다.

```tsx
<Routes>
  <Route path="/" element={<Home />} />
  <Route path="/about" element={<About />} />
</Routes>
```

이 모양이 코드를 읽는 내 관점에서 직관적이라 이해하기 쉬웠다. 그런데 React Router 공식 문서는 **Data Router API**를 쓸 수 있는 `createBrowserRouter` + `RouterProvider`를 권장하고 있었다.

Data Router API는 `loader`, `action` 같은 걸로 **URL이 들어오는 순간 데이터를 먼저 가져오게** 해준다. 기존에 컴포넌트가 렌더링된 _후_ `useEffect`에서 데이터를 패칭하던 것과 시점이 다르다.

그래서 생긴 질문:

> **왜 `RouterProvider`는 이게 되고, `BrowserRouter`는 안 될까? 그래서 새 방식으로 마이그레이션을 권장하는 걸까?**

이 질문 하나를 끝까지 따라간 게 이 글이다.

---

## Session 1. 두 방식의 사용법

### BrowserRouter

`Routes` 안에 `Route`를 정의해 URL 경로와 컴포넌트를 연결한다. 라우트가 **JSX 컴포넌트 트리 안에** 들어있다.

```tsx
function App() {
  return (
    <Routes>
      <Route path="/" element={<Home />} />
      <Route path="/about" element={<About />} />
    </Routes>
  );
}
```

### RouterProvider

`main.tsx`에서 `RouterProvider`를 선언하고, 라우트 정의는 별도 파일에서 `createBrowserRouter()`로 만들어 import해온다. 라우트가 **객체 배열로** 정의된다.

```tsx
// main.tsx
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import "./index.css";
import { RouterProvider } from "react-router-dom";
import { routers } from "./routers";

createRoot(document.getElementById("root")!).render(
  <StrictMode>
    <RouterProvider router={routers} />
  </StrictMode>
);
```

```tsx
// routers.ts
import { createBrowserRouter } from "react-router-dom";
import { App } from "./App";
import { HomePage } from "./pages/HomePage";

export const routers = createBrowserRouter([
  {
    path: "/",
    element: <App />,
    children: [{ index: true, element: <HomePage /> }],
  },
]);
```

여기서 `path`는 URL 경로, `element`는 그릴 컴포넌트, `children`은 하위 라우트다. 부모 `element`(`<App />`)는 레이아웃 역할을 하고, 그 안의 `<Outlet />` 자리에 자식이 끼워져 렌더링된다.

## Session 2. 핵심 차이 — "라우트가 언제 정해지나"

한 줄 요약:

> **BrowserRouter는 라우트를 JSX 안에 선언해서 라우트 정보를 "렌더링 단계에" 알게 된다.
> RouterProvider는 라우트를 "렌더링 전에" 객체로 미리 정의한다.**

왜 이런 차이가 나는지가 이 글의 본론이다. 그리고 그 답은 React가 아니라 **JavaScript가 코드를 언제 실행하느냐**에 있었다.

### createBrowserRouter는 "함수 호출"이다

```tsx
export const routers = createBrowserRouter([...]);
```

`createBrowserRouter([...])`는 `()`가 붙은 **함수 호출**이다. JS는 코드를 위에서 아래로 읽다가 호출문을 만나면 그 자리에서 바로 실행한다. 그래서 이 줄을 만나는 순간 함수가 실행되고, 라우트 정보를 전부 분석한 **라우터 객체**를 돌려준다.

### BrowserRouter의 `<Route>`는 "JSX"다

반면 `<Route />` 같은 JSX는 실행되는 게 아니라 **객체로 변환만** 된다. 이건 다음 세션에서 자세히 본다. 핵심은 — JSX는 객체(설계도)로 남아있다가 **렌더링 단계가 와야** 비로소 의미를 갖는다는 것.

## Session 3. JSX는 객체로 변환된다

`<Route />`가 어떻게 처리되는지 보자.

```tsx
<Route path="/" element={<Home />} />
```

이 JSX는 빌드 타임(Babel/번들러)에 이렇게 변환된다.

```js
React.createElement(Route, { path: "/", element: ... })
```

이름이 `document.createElement`랑 비슷해서 "실제 DOM을 만드나?" 싶지만, **전혀 무관하다.** `React.createElement`는 진짜 DOM 노드가 아니라 **React Element라는 객체**를 만들 뿐이다.

|           | React.createElement             | document.createElement |
| --------- | ------------------------------- | ---------------------- |
| 결과물    | `{ type, props }` 객체 (설계도) | 실제 DOM 노드          |
| 화면 반영 | 안 됨 (나중에 React가)          | 즉시                   |
| 성격      | 선언적                          | 명령형                 |

그래서 위 코드는 이런 객체를 만든다. (이게 VDOM의 한 조각이다)

```js
{ type: Route, props: { path: "/", element: ... } }
```

**객체로 만드는 이 시점엔 아무것도 실행되지 않는다.** 그냥 JSX를 풀어서 객체(설계도)로 만들 뿐이다. 이 설계도가 의미를 갖는 건 **렌더링 단계**다.

> ### 짚고 넘어갈 오해 하나
>
> "JSX라서 나중에 실행된다"는 절반만 맞다. `main.tsx`의 `<RouterProvider />`도 JSX인데, 이건 `render()`의 인자로 **top-level에** 있어서 모듈 평가 때 바로 객체로 만들어진다. 즉 **JSX냐 아니냐가 시점을 가르는 게 아니라, top-level이냐 / 함수 안에 갇혔냐가 가른다.** 컴포넌트 함수 안의 JSX가 나중에 실행되는 건 "JSX라서"가 아니라 **"함수 안에 있어서, 그 함수가 렌더 단계에 호출되기 때문"**이다.

## Session 4. 그럼 렌더링은 언제 시작되나

코드가 렌더링되기 시작하는 신호는 딱 하나다.

```tsx
createRoot(...).render(<RouterProvider router={routers} />);
```

이 `render()` 호출이 **React에게 일을 넘기는 신호**다. 그전까지는 전부 평범한 JS 실행일 뿐, React(Fiber)는 일을 시작하지도 않았다.

RouterProvider의 대략적인 흐름:

1. `main.tsx`가 `routers.ts`를 import
2. `routers.ts`의 `createBrowserRouter()`가 즉시 호출됨 → 라우터 객체 완성
3. `main.tsx`의 `createRoot().render()` 실행 → 이후 렌더링 시작
4. React가 렌더링

여기서 2번이 3번보다 **먼저** 일어난다는 게 핵심이다. 라우터가 렌더링 시작 전에 이미 완성돼 있다.

## Session 5. 모듈 평가 — JS가 코드를 실행하기 전에 하는 일

> 참고: 흔히 "모듈 로딩 단계"라고 부르지만, ES 명세상 정확한 이름은 **모듈 평가(evaluation)**다. 모듈은 construction → linking → evaluation 순으로 처리된다.

### 1단계 [연결] — 평가 전에 그래프부터 그린다

ES 모듈은 코드를 실행(평가)하기 **전에**, import를 전부 훑어서 의존성 그래프부터 그린다. `main.tsx` 것만 연결하는 게 아니라 **전체 의존성 트리를 다 따라간다.**

```
main.tsx
├─ "react"
├─ "react-dom/client"
├─ "./index.css"
├─ "react-router-dom"
└─ "./routers"
   ├─ "react-router-dom"   (이미 연결됨 → 재사용)
   ├─ "./App"
   └─ "./pages/HomePage"
→ 이 트리 전체를 먼저 다 연결
```

> 이건 JS 실행 컨텍스트의 호이스팅과 발상이 닮았다. "실행 전에 미리 코드를 훑어 준비해두는 단계"라는 점에서. 단, 둘은 다른 레이어다 — 모듈 그래프는 **파일 간** 연결(모듈 시스템)이고, 호이스팅은 **실행 컨텍스트 안**의 선언 등록(JS 엔진)이다. 시점상으론 모듈 그래프 연결이 먼저고, 그 안에서 각 모듈이 평가될 때 호이스팅이 따로 일어난다.

### 2단계 [평가] — 의존성 깊은 곳부터 실행

그래프를 다 그린 뒤, 의존성 깊은 것부터(post-order) 위로 평가한다. 그래서 `main.tsx`보다 `routers.ts`가 **먼저** 평가된다.

`routers.ts`를 평가할 때, 파일 최상단(top-level)에 `createBrowserRouter([...])` 호출문이 있으니 즉시 실행되어 `routers` 객체가 완성된다.

```tsx
// routers.ts
import { createBrowserRouter } from "react-router-dom";

export const routers = createBrowserRouter([...]);
//          ↑ 어떤 함수 안에도 안 갇힌 top-level + () 호출문
//            → 모듈 평가 때 즉시 실행
```

### top-level vs 함수 안 — 실행 시점을 가르는 진짜 기준

```tsx
function App() {
  const data = computeSomething(); // ← 함수 안에 있음
  return <div>{data}</div>;
}
```

`computeSomething()`은 `App` 함수 **안에** 있다. 이건 모듈 평가 때 실행되지 **않는다.** `App`이 아직 호출되지 않았기 때문이다. 이 코드는 그저 "정의"만 된 상태고, `App`이 렌더 단계에서 React에 의해 호출돼야 비로소 `computeSomething()`도 실행된다.

**그래서 라우터(`createBrowserRouter`)는 top-level이라 렌더링 전에 완성되고, 컴포넌트 안의 코드는 렌더 단계로 미뤄진다.**

## Session 6. 전체 타임라인 (SPA / CSR)

지금까지를 하나의 타임라인으로 합치면 이렇게 된다.

```
[0. 빌드 타임 — Babel/번들러, 브라우저 실행 전]
   모든 JSX <...> → React.createElement(...) 호출문으로 변환 완료
   (이 시점 이후 코드에 JSX는 없음. 전부 함수 호출문)

[1. 모듈 평가 — JS 엔진, render() 호출 전]
   (a) import 그래프 전체 연결 → 의존성 깊은 것부터 평가
   (b) routers.ts 평가:
       createBrowserRouter([...]) 실행 ← top-level 호출문, 즉시 실행
       → routers 객체 완성 ✅
   (c) App.tsx, HomePage.tsx 평가:
       function App() {...} "정의"만 등록 (실행 X)
       ★ App 함수 안의 createElement는 아직 실행 안 됨
   (d) main.tsx 평가 (위→아래로):
       - createElement(StrictMode, ..., createElement(RouterProvider, {router: routers}))
         ← top-level 인자라 여기서 실행됨 → 엘리먼트 객체 생성
       - createRoot(...).render(그 객체) 호출 ← React에게 일을 넘기는 신호

──────── 여기서부터 React(Fiber)의 작업 시작 ────────

[2. 렌더 단계 — Render Phase]
   (e) React가 RouterProvider 렌더 → 현재 URL("/")과 라우트 매칭
   (f) 매칭된 App을 React가 호출: App()
       ★ 이때 비로소 App 안의 createElement("div"...) 실행 → 엘리먼트 객체 생성
   (g) 자식(HomePage)도 같은 식으로 호출되어 엘리먼트 객체 생성
   (h) 엘리먼트 객체들을 모아 VDOM 트리 구성
   (i) 리콘실리에이션: 이전 VDOM과 비교 (첫 렌더는 비교 대상 없어 전체 생성)

[3. 커밋 단계 — Commit Phase]
   (j) 계산된 변경사항을 실제 DOM에 반영 → 화면에 그려짐

──────── 화면이 그려진 "후" ────────

[useEffect 실행]
   등록해뒀던 콜백이 이제 실행됨
   → fetch() 시작 → (네트워크 대기) → 데이터 도착
   → setState → 리렌더 → 다시 커밋 → 화면 업데이트
```

> ### VDOM을 왜 거치나?
>
> 실제 DOM 조작은 비싸다. 그래서 가벼운 객체(VDOM)로 먼저 그려두고, 이전 VDOM과 비교(diff)해서 **바뀐 부분만** 실제 DOM에 반영한다. 이게 React가 빠른 이유다.

## Session 7. useEffect의 fetch는 "가장 나중"에 실행된다

### `useEffect(() => {...}, [])`를 "만나는" 것과 "실행"은 별개다

렌더 단계에서 `App` 함수가 실행되며 `useEffect` 줄을 만나긴 한다. 하지만 그건 **"이 콜백을 나중에 실행해줘"라고 React에 등록(예약)**하는 것이지, 콜백을 그 자리에서 실행하는 게 아니다. `createElement`가 함수를 객체에 담아만 두던 것과 같은 결이다 — 콜백을 `useEffect`에 넘겨 **보관**하는 것.

그래서 fetch는 렌더 중에 안 돈다. React가 **커밋(화면 그리기)을 끝낸 뒤에** 모아뒀던 effect 콜백들을 실행한다.

### 왜 굳이 화면 그린 "후"인가

렌더 단계는 "순수"해야 한다(같은 입력이면 같은 출력, 부수효과 없음). fetch 같은 부수효과(side effect)를 렌더 도중에 하면 렌더가 더러워진다. 그래서 React는 부수효과를 렌더 바깥, **커밋 이후로** 밀어낸다. 이름이 `useEffect`(effect = 부수효과)인 이유다.

### 전체 순서

```
1. createBrowserRouter([...])          ← 모듈 평가 (가장 먼저)
2. <RouterProvider/> 객체 생성          ← 모듈 평가
3. render() 호출 → React 시작
4. App() 호출, useEffect "등록"          ← 렌더 단계 (콜백 실행은 아직 X)
5. 실제 DOM 반영, 화면 그려짐            ← 커밋 단계
6. useEffect 콜백 실행 → fetch 시작     ← 커밋 "이후" (가장 나중) ★
7. 데이터 도착 → setState → 4~5 다시
```

## Session 8. 그래서 BrowserRouter엔 워터폴 문제가 생긴다

`useEffect`로 데이터를 패칭하면 이런 순서가 된다.

```
useEffect 방식:
  렌더 → 커밋 → 화면(데이터 없는 빈/로딩 상태) → 그제서야 fetch 시작
                                              → 데이터 도착 → 다시 렌더
```

화면을 **일단 그린 다음에** fetch가 시작되니, 사용자는 빈 화면이나 스피너를 먼저 본다. 게다가 중첩 컴포넌트면 부모 렌더 → 부모 fetch → 자식 렌더 → 자식 fetch... 순으로 **줄줄이 늦어진다.** 이게 워터폴이다.

반면 `loader`는 데이터를 렌더보다 먼저 가져온다.

```
[loader 방식]
  URL 변경 감지
     ↓
  loader 실행 → fetch        ★ 렌더링 "전"
     ↓ (데이터 도착까지 대기)
  데이터 준비됨
     ↓
  컴포넌트 렌더 → 커밋 → 화면 (이미 데이터 들고 그려짐)

[useEffect 방식]
  컴포넌트 렌더 → 커밋 → 화면 (데이터 없이 먼저 그려짐)
     ↓
  useEffect 콜백 실행 → fetch  ★ 렌더링(커밋) "후"
     ↓ (데이터 도착)
  setState → 다시 렌더 → 커밋 → 화면 업데이트
```

**loader는 렌더 "전"에, useEffect의 fetch는 커밋 "후"에 실행된다.** fetch가 렌더링의 앞에 서느냐 뒤에 서느냐의 차이다.

### 실제로 뭐가 달라지나

- **사용자가 보는 화면**: loader는 데이터 들고 한 번에 그려져 로딩 깜빡임이 적다. useEffect는 빈 화면이 먼저 뜨고 나중에 채워진다.
- **렌더 횟수**: loader는 개념상 1번. useEffect는 빈 상태 1번 + 데이터 도착 후 1번 = 최소 2번.
- **워터폴(중첩일 때)**: loader는 매칭된 라우트들의 loader를 **병렬로** 실행. useEffect는 부모→자식 순차라 줄줄이 늦어진다.

> 이게 바로 "Data Router API의 장점"의 실제 메커니즘이다. 그리고 이걸 가능하게 한 건 Session 5에서 본 **"라우터가 렌더링 전에 객체로 완성된다"**는 그 성질이다. 라우터가 렌더 전에 라우트 구조를 다 알고 있으니, 렌더 전에 해당 라우트의 loader를 먼저 돌릴 수 있는 것.

---

## Session 9. 인사이트 — loader / 서버 컴포넌트 / useEffect는 한 축에 있다

이 흐름을 보다가, 예전에 Next.js에서 **서버 컴포넌트로 데이터를 미리 가져오게** 했던 것과 발상이 똑같다는 걸 깨달았다. 셋 다 **"데이터 패칭을 렌더링보다 앞으로 당긴다"**는 핵심 아이디어를 공유한다.

```
useEffect:     렌더 → 커밋 → 화면 → 그제서야 fetch    (데이터가 뒤)
loader:        fetch → 데이터 준비 → 렌더 → 화면       (데이터가 앞)
서버 컴포넌트:  서버에서 fetch → 데이터 채워 렌더 → HTML 전송   (데이터가 앞)
```

단, loader와 서버 컴포넌트는 **실행 장소가 다르다.** loader는 여전히 브라우저(클라이언트)에서 렌더 전에 fetch하는 것이고, 서버 컴포넌트는 아예 **서버에서** fetch와 렌더를 끝내 HTML로 내려준다. "데이터를 앞으로 당긴다"는 같지만, "어디서 당기느냐"가 다른 셈.

(흥미롭게도 React Router의 이 데이터 API는 원래 Remix라는 서버 프레임워크에서 가져온 것이라, 계보가 실제로 이어져 있다.)

### 결론

> 가장 큰 차이는 **"렌더링 전에 객체로 정의한다(RouterProvider)" vs "렌더링 도중에 라우트를 알게 된다(BrowserRouter)"**이다.
>
> 렌더링 전에 라우트를 객체로 정의해두면, 데이터를 가져오는 단계를 `useEffect`(커밋 후)보다 **더 일찍** 진행할 수 있어 워터폴 문제를 개선할 수 있다. 이것이 공식 문서가 `RouterProvider`를 권장하는 핵심 이유다.
>
> 다만 데이터 기능을 안 쓴다면 `BrowserRouter`로도 충분하며, 공식 문서도 이를 인정한다. (실제로 캐싱·백그라운드 갱신 같은 영역은 loader가 약해서, 현업에선 React Query 같은 도구를 함께 쓰는 경우가 많다.)

## 부록. 면접 대비 Q&A

### 1단계 — 기본

**Q. BrowserRouter 대신 createBrowserRouter + RouterProvider를 쓴 이유는?**
→ v6.4(2022.9)에서 데이터 API와 함께 도입됨. 라우트를 렌더링 전에 객체로 정의해 loader 등을 쓸 수 있어서. 단, 데이터 기능을 안 쓰면 BrowserRouter도 괜찮다는 점까지 언급하면 균형 잡힌 답.

**Q. 둘의 구조적 차이는?**
→ BrowserRouter는 라우트를 JSX로 선언(렌더 중 결정), createBrowserRouter는 객체로 선언(렌더 전 결정).

**Q. path / element / children / Outlet의 역할은?**
→ element는 레이아웃, children은 하위 라우트, Outlet은 자식이 끼워지는 자리. 부모는 유지되고 Outlet 안만 교체됨.

**Q. index: true와 Navigate replace는 왜 썼나?**
→ index는 부모 경로의 기본 자식. replace는 히스토리 교체라 뒤로가기 무한 루프를 방지.

### 2단계 — 원리

**Q. "렌더링 전에 라우트가 정의된다"가 어떻게 가능한가?**
→ createBrowserRouter는 `()` 붙은 함수 호출이라 top-level에서 모듈 평가 때 즉시 실행. JSX는 객체로 변환만 되고 렌더 시점까지 실행이 미뤄짐.

**Q. JSX는 어떻게 동작하나? 왜 React가 JSX를 쓰나?**
→ 빌드 타임에 `React.createElement()`로 변환 → `{type, props}` 객체(React Element) 생성 → 모여서 VDOM. JSX는 그 객체를 편하게 쓰는 문법.

**Q. React.createElement와 document.createElement 차이는?**
→ 전자는 설계도 객체 반환, 후자는 실제 DOM 노드 즉시 생성. 선언적 vs 명령형.

**Q. 함수가 즉시 실행되는 기준은?**
→ JSX냐 아니냐가 아니라, top-level 호출문이냐 / 함수 안에 갇혔냐. top-level이면 모듈 평가 때, 함수 안이면 그 함수 호출 시점.

### 3단계 — 깊이

**Q. 모듈 평가 단계와 실행 컨텍스트(호이스팅)는 같은 건가?**
→ 다른 레이어. "실행 전 미리 준비"라는 발상은 닮았지만, 모듈 시스템(파일 간) vs JS 엔진(컨텍스트 안)이고 포함 관계.

**Q. useEffect 안의 fetch는 언제 실행되나?**
→ 렌더 단계에선 콜백 "등록"만, 실제 실행은 커밋(화면 그린) 후. 그래서 로딩 깜빡임/워터폴 발생.

**Q. loader와 useEffect fetch의 시점 차이는?**
→ loader는 렌더 전, useEffect는 커밋 후. loader가 워터폴에 유리.

**Q. 그럼 loader가 무조건 좋은가?**
→ 아니다. 캐싱·백그라운드 갱신 등은 약해서 React Query 같은 도구가 그 자리를 채움. 트레이드오프 인지가 포인트.

**Q. VDOM을 굳이 거치는 이유는?**
→ 실제 DOM 조작은 비싸서, 가벼운 객체로 먼저 그리고 이전 VDOM과 비교(diff)해 바뀐 부분만 반영.

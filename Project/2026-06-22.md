### 0. 코테

코테에서 함수를 사용하는방법을 알게 됨.

코테에서도 추상화를 이용하자

**메인의 큰 틀이 보이도록 함수로 작성해서 큰 줄기가 보이도록 하기**

**여기서 Intact, isIntact를 통해 내부 상세함수는 숨기고 큰 틀이 보이도록 함.**

딱봐도 온전수면, 카운트한다는 함수인지 보인다.

```jsx
#온전수 계산하는 함수
def Intact(a,b) :
    cnt=0

    for i in range(a,b+1) :
        if isIntact(i) :
            cnt+=1
    print(cnt)

print(Intact(a,b))
```

**프로젝트에서도 이렇게 사용하면 좋을뿐더러 코테에서도 활용하자**

이렇게 코테 작성하는 법은 전에 인프랩에서 코테 강의에서도 봤던 방식이다.

**핵심이 드러나게 작성하기!**

```jsx
a, b = map(int, input().split())

cnt=0

#모든 자릿수의 합이 짝수인 수 구하는 함수
def isSumF(i):
    sum_val = sum(map(int,list(str(i))))
    if sum_val%2==0:
        return True
    else :
        return False

#소수
def isMinority(i) :
    for ele in range(2,i) :
        if i%ele==0 :
            return False

    return True

for i in range(a,b+1) :
    if isMinority(i) and isSumF(i) :
        cnt+=1

print(cnt)
```

## 1. 윈도잉(windowing) 기법

### 개념

윈도잉은 **전체 데이터 집합을 한꺼번에 처리하지 않고, 현재 관심 있는 일부 구간(window)만 선택해 다루는 기법**이다.

실제 창문을 떠올리면 쉽다. 창문 밖 모든 배경이 한눈에 보이는 게 아니라, 창틀에 걸친 구간만 보인다. 윈도잉도 마찬가지로 불필요한 부분은 가리고 필요한 부분만 보이게 한다. 처음 다루는 양은 전체의 아주 작은 일부분이며, 이 구간을 데이터 위에서 이동시키면서 보이지 않게 된 구간은 새로 보이는 구간과 교체·재활용한다.

핵심 발상은 한 문장으로 압축된다.

> **전체가 있는 것처럼 동작하되, 실제로는 일부 구간만 활성 상태로 둔다.**

### 윈도잉은 프론트엔드만의 개념이 아니다

이번에 알게 된 점. 이 발상은 여러 분야에서 독립적으로 반복 등장한다.

- **UI 렌더링** : 가상 스크롤. window는 화면에 보이는 영역. 목적은 DOM/메모리 절약.
- **스트림 처리** (Kafka, Flink 등) — 무한 이벤트를 시간/개수 단위로 자르는 tumbling/sliding/session window.
- **네트워크 (TCP)** : 슬라이딩 윈도우. 한 번에 보낼 수 있는 데이터 양을 window로 제한하고, 확인되면 앞으로 민다.
- **알고리즘** : 슬라이딩 윈도우 기법. 연속 구간을 한 칸씩 옮겨가며 매번 전체 재계산을 피해 O(n)으로 떨어뜨린다.
- **신호 처리** : 긴 신호를 짧은 구간으로 잘라 분석하는 윈도우 함수(Hann, Hamming 등).

이들이 하나의 공식 용어에서 파생된 건 아니다. 비슷한 문제(크거나 무한한 데이터를 부분 구간으로 다루기)에 대해 각 분야가 독립적으로 "window"라는 같은 비유를 채택한 쪽에 가깝다. 그래서 "같은 뿌리"보다 **"공통된 발상을 공유한다"** 고 표현하는 게 정확하다.

## 2. 가상 스크롤

### 왜 필요한가

무한 스크롤로 아이템을 계속 불러오다 보면 많게는 10,000개까지 데이터가 쌓이는 경우가 있다. 실제로 리스트에 10,000개를 그대로 넣어봤더니, 화면 이동이 느려지고 상호작용할 때 전반적으로 끊김이 발생했다.

원인은 단순했다. **10,000개를 모두 DOM에 올려두었기 때문**이다. 브라우저가 이 모든 아이템을 들고 있어야 하니 무거워질 수밖에 없다.

수치로 확인해보면 차이가 분명하다.

| 비교 항목    | Naive rendering         | react-window                   |
| ------------ | ----------------------- | ------------------------------ |
| DOM node 수  | 매우 많음 (1만 개 전부) | 화면에 보이는 만큼만 (수십 개) |
| JS heap size | 큼                      | 작음                           |

DOM에 전부 올린 쪽은 메모리와 노드 수를 엄청나게 차지하는 반면, react-window 쪽은 DOM에 올라와 있는 노드가 몇 개 되지 않았다.

불필요한 아이템을 제거하면 된다. 브라우저에서 불필요한 건 **지금 화면에 보이지 않는 부분**이다.
현재 보고 있는 화면 안에서, 필요한 만큼만 렌더링하면 어떨까? 바로 이 방법을 사용한 기술이 **가상 스크롤**이다.

### 정의

가상 스크롤은 **윈도잉을 UI 렌더링에 적용한 사례**다. 전체 데이터 중 현재 화면(viewport)에 보이는 일부만 실제로 DOM에 렌더링하여, 대량의 리스트를 적은 비용으로 처리하는 최적화 기법이다.

핵심은 두 가지가 한 쌍으로 묶인다는 점이다.

1. **보이는 범위의 아이템만 렌더링한다.** 화면 밖으로 벗어난 항목의 DOM 노드는 제거되거나 새 항목으로 재사용된다.
2. **전체 콘텐츠의 높이는 실제 항목 수만큼 미리 확보해둔다.** 그래야 스크롤바가 1만 개가 들어있는 것처럼 정상 동작한다.

이 둘 중 하나만 알면 정의가 헐거워진다. "보이는 것만 그린다"만 말하면 "그럼 스크롤바는?"이라는 의문이 남기 때문이다. 우리가 보고 있는 화면이 창문이라면, 보이지 않는 부분은 가리고 보이는 부분만 창문을 통해 보이도록 구현하는 것이다.

## 3. 동작 원리 (react-window)

### API: FixedSizeList

고정 높이 리스트는 `FixedSizeList`를 쓴다. 핵심 4개 prop만 있으면 동작한다.

```jsx
import { FixedSizeList } from "react-window";

const items = Array.from({ length: 10000 }, (_, i) => `Item ${i}`);

const Row = ({ index, style }) => (
  <article style={style}>{items[index]}</article>
);

const ListComponent = () => (
  <FixedSizeList
    height={500} // 보이는 영역(viewport)의 높이 (px)
    width={500} // 너비 (px)
    itemSize={120} // 각 행의 고정 높이 (px)
    itemCount={items.length} // 전체 아이템 개수
  >
    {Row}
  </FixedSizeList>
);
```

추가 옵션 두 가지도 자주 쓴다.

- `itemData` : item 전체 데이터를 명시적으로 넘기면 Row의 render prop에서 `data`로 받을 수 있다. 클로저 참조보다 깔끔하고 메모이제이션에 유리하다.
- `overscanCount` : 보이는 범위 앞뒤로 여분을 미리 렌더링해 스크롤 시 빈 공간이 깜빡이는 걸 막는다. 단, 너무 크면 성능에 역효과라 보통 2~5면 충분하다.

컨테이너 크기에 맞춰 width/height를 자동으로 잡고 싶다면 `react-virtualized-auto-sizer`를 함께 쓴다.

### 가장 헷갈렸던 부분: Row 함수

`Row`는 특별한 게 아니라 **"index를 받아서 그 번째 행을 그려주는 함수"** 일 뿐이다. 그리고 가장 중요한 사실:

**Row는 10,000번 호출되지 않는다.** 화면에 보이는 만큼 + overscan만큼만 호출된다.

`index`, `style`, `data`는 우리가 넣는 게 아니라 **react-window가 넣어준다.** 우리는 "이런 게 들어올 테니 받아서 쓸게" 하고 함수만 정의해 자식으로 넘긴다(render prop 패턴).

### style

react-window가 각 행마다 "너는 화면의 이 위치에 있어야 해"를 계산해서 `style`에 담아준다. 예를 들어 7번 행(itemSize 72px)이라면:

```jsx
style = {
  position: "absolute",
  top: 504, // 7 × 72
  left: 0,
  width: "100%",
  height: 72,
};
```

그래서 `<article style={style}>`로 이 style을 **최상위 요소에** 그대로 적용해야 한다. 빼먹으면 모든 행이 위치 정보 없이 0번 위치에 전부 겹쳐버려서, 스크롤할 게 아무것도 없게 된다. 우리가 좌표를 계산할 필요는 전혀 없다. react-window가 다 해주고 우리는 받은 style을 붙이기만 하면 된다.

### react-window는 스크롤 위치를 어떻게 계산할까?

우리 코드에는 스크롤 위치를 계산하는 코드가 **없다.** 전부 `FixedSizeList` 내부에서 일어나고, 우리는 재료(`itemSize`, `height`, `itemCount`)만 넘긴다. 내부 동작을 따라가 보면 이렇다. (itemSize 72, height 560 기준)

**1. 스크롤 이벤트 받기** — 내부에 스크롤 가능한 컨테이너를 만들고 `onScroll`을 붙인다. 사용자가 스크롤하면 브라우저가 `scrollTop`(맨 위에서 내려간 px)을 준다. 이건 계산이 아니라 브라우저가 주는 값이다.

**2. 첫 보이는 index 구하기** — 모든 행 높이가 같으니 나눗셈 한 번이면 끝.

```jsx
startIndex = Math.floor(scrollTop / itemSize)
           = Math.floor(1000 / 72)
           = 13
```

**3. 마지막 보이는 index 구하기** — viewport 높이를 행 높이로 나누면 보이는 개수가 나온다.

```jsx
보이는 개수 = Math.ceil(560 / 72) = 8
endIndex = startIndex + 8 = 21 (대략)
```

**4. overscan 더하기** — `overscanCount={5}`면 위아래로 5개씩 추가.

```jsx
실제 렌더 범위 = (13 - 5) ~ (21 + 5) = 8번 ~ 26번 (약 19개)
```

**5. 각 행의 top 계산** — 곱셈 한 번.

```jsx
15번 행의 top = index × itemSize = 15 × 72 = 1080
→ style = { position: "absolute", top: 1080, height: 72, ... }
```

결국 스크롤 위치 계산의 정체는 **두 줄의 산수**다.

```jsx
startIndex   = Math.floor(scrollTop / itemSize)   // 어디서부터 보이는지
보이는 개수   = Math.ceil(height / itemSize)        // 몇 개가 보이는지
```

### 고정 높이가 빠른 이유 (vs 가변 높이)

모든 행 높이가 같으면 `index × itemSize`로 위치를 즉시 알 수 있고, 역으로 `scrollTop ÷ itemSize`로 index를 즉시 구할 수 있다. 곱셈·나눗셈뿐이라 1만 개든 100만 개든 계산 비용이 거의 일정하다.

반면 행 높이가 제각각인 `VariableSizeList`는 이 나눗셈이 안 통한다. 각 행의 누적 높이를 따로 저장해두고 "scrollTop이 어느 행 구간에 속하는지"를 탐색해야 해서 계산이 더 복잡하다. 고정 높이를 쓸 수 있다면 가장 빠른 경로를 타는 셈이다.

### 실제 적용한 코드 메모

데모에서 현재 렌더된 개수를 보여주려고 `onItemsRendered`를 썼다.

```jsx
<FixedSizeList
  height={560}
  width="100%"
  itemCount={items.length}
  itemSize={72}
  itemData={items}
  overscanCount={5}
  onItemsRendered={({ visibleStartIndex, visibleStopIndex }) =>
    setRenderedCount(visibleStopIndex - visibleStartIndex + 1)
  }
>
  {Row}
</FixedSizeList>
```

여기서 `visibleStartIndex` / `visibleStopIndex`는 overscan을 **뺀** "실제 화면에 보이는" 범위다. 그래서 미리 그려둔 overscan 5개씩은 카운트에서 빠진다.

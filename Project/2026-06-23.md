# 코딩테스트

## 불변성 vs 가변성

파이썬에서 immutable(불변성), mutable(가변성)을 가진 속성이 나뉘어진다고 했다. 처음알았다 … 다 가변성인줄알았따.

- **immutable(불변성) : tuple, string, int, bool ⇒ Primitive type : 원시타입**
- **mutable(가변성) : list, dict**

즉, immutable 불변성을 지닌 타입이 인자로 넘어가면, **값을 복사해서 넘긴다**더라. → 본인 자체가 아닌 복사본이다!

그래서 아래 코드 처럼 함수 안에 넘어간 원시타입은 불변성때문에, return 이 아닌 이상 값이 변하지 않는다.

> **불변성 타입은 함수 인자로 넘어가면, 복사본으로 넘어간다.**

그래서 외부에 영향을 주지 않는다.

```jsx
//잘못된 함수
def swap(a, b):
    a, b = b, a
    print(a, b)
    return a,b

n, m = 10, 20
swap(n, m)
print(n, m)

```

list는 가변성 타입이기 때문에 함수안에서 일어난 일에 대해 영향을 받는다(사이드 이펙트 발동)

> **가변성 타입은 함수 인자로 넘어가면, 변할 수 있다.**

아래 코드는 변후 modify 함수 안에서 함수가 변하도록 지시하였음에, 외부에서 영향을 받았다.

```jsx
def modify(arr):
    arr[0] = 10

_list = [1, 2, 3, 4]
modify(_list) //modify(_list[:]) : 사이드 이펙트 없애기

for elem in _list:
    print(elem, end=" ")

```

- 부수효과를 없애기 위해서는, 함수안에 리스트 값을 복제(슬라이싱) 해서 넘기면된다. ← primitive 타입처럼 복사본 보내기

## 전역변수 global

```jsx
product = 1

def f(n):
    for i in range(1, n + 1):
        product *= i

f(5)

print(product)
```

product가 전역변수라서 되는 줄알았지만, **읽을 경우에만 전역변수를 가져올수있고, 접근할경우 global이라고 칭해줘야된다. 위는 에러가 발생하는 코드이다 product가 지역변수로 들어가기 때문.**

```jsx
product = 1
def f(n):
    global product
    for i in range(1, n + 1):
        product *= i
f(5)
print(product)  # 120
```

# 프로젝트 : 동적 높이 가상 스크롤 (react-window VariableSizeList)

## 1. 고정 높이가 빠른 이유

고정 높이의 모든 계산은 곱셈/나눗셈 한 번이다.

```jsx
startIndex = Math.floor(scrollTop / itemSize)   // 스크롤 위치 → index
top        = index × itemSize                    // index → 위치
전체 높이   = itemCount × itemSize                // 스크롤바 높이
```

이게 가능했던 건 **모든 행 높이가 같다**는 단 하나의 전제 덕분이다.

높이가 일정하면 index와 위치(px)가 정비례하니, 둘 사이를 곱셈/나눗셈으로 즉시 오갈 수 있다.

앞 행들을 더할 필요 없이 `index × itemSize`로 위치가 바로 나온다. **O(1)**.

## 2. 동적 높이가 되면 이 전제가 무너진다

행 높이가 제각각(60, 120, 80, 200...)이 되는 순간 정비례가 깨지고, 위 세 계산이 전부 무너진다.

- **문제 1 위치 계산**: `7 × itemSize`가 안 된다.
  - 0~6번 높이를 전부 더해야 한다 → 누적 덧셈(최적/최선의 시복 O(n)).
- **문제 2 시작 index 탐색**: `scrollTop ÷ itemSize`가 안 된다.
  - 높이를 하나씩 누적하며 scrollTop을 넘는 지점을 찾아야 한다 → 선형 탐색 O(n).
- **문제 3 전체 높이**: `itemCount × itemSize`가 안 된다. 모든 행 높이를 알아야 한다.

### 가장 큰 문제점 : 그려보기 전에는 높이를 모른다

문제 1~3은 **높이를 다 안다면** 누적합으로 풀 수 있다.

**각 행의 실제 높이는 DOM에 그려보기 전엔 알 수 없다.** 내용이 길면 줄바꿈되고, 이미지가 늦게 로드되면 또 바뀐다. 여기서 순환이 생긴다.

위치를 계산하려면 → 높이를 알아야 하고 → 높이를 알려면 → 그려봐야 하고 → 가상 스크롤은 보이는 것만 그리려고 → 위치를 먼저 계산하려 한다 → (다시 처음)

안 그린 행의 높이는 영영 모른다. 고정 높이에선 높이가 상수라 이 순환이 아예 없었다.

### 순환을 깨는 개선점

- **추정 후 측정,보정하기** : 일단 **추정치**로 그린 뒤 실제 높이를 측정해 보정. (TanStack Virtual, react-virtuoso)
- **높이를 미리 알려주기** : 개발자가 `itemSize`를 `(index) => 높이` 함수로 제공. 그려봐야 안다는 고리를 개발자 지식으로 끊는다. (react-window VariableSizeList)

react-window는 **측정하지 않는다. 개발자가 알려준 값을 믿는다.**

## 3. VariableSizeList

```jsx
import { VariableSizeList } from "react-window";

const Row = ({ index, style }) => <div style={style}>{items[index].text}</div>;

const getItemSize = (index) => {
  // items[index]에 대한 높이를 반환 (개발자가 책임)
};

const ListComponent = () => (
  <VariableSizeList
    height={500}
    width={500}
    itemCount={items.length}
    itemSize={getItemSize} // 숫자가 아니라 함수
    estimatedItemSize={100} // 안 가본 영역 추정용
  >
    {Row}
  </VariableSizeList>
);
```

핵심 장치는 세 가지이다.

### 장치 1: 누적 오프셋 캐시 (문제 1 해결)

각 행의 높이가 아니라 **각 행이 시작하는 위치(offset)** 를 **누적합**으로 계산해 저장한다.

```
높이:    [ 60, 120,  80, 200,  95, ...]
offset:  [  0,  60, 180, 260, 460, ...]   ← 각 행의 시작 위치
```

이 offset 배열이 동적 높이의 심장이다. `offset[n] = n번 행의 top`이라 위치 계산이 **다시 O(1)** 이 된다. 매번 더하는 게 아니라 한 번 계산해 저장하고 재사용한다.

### 장치 2: 이진 탐색 (문제 2 해결)

scrollTop이 300px일 때 시작 index를 찾으려면 offset에서 "300이 들어가는 구간"을 찾아야 한다. offset이 `[0, 60, 180, 260, 460]`이면 300은 260(3번)과 460(4번) 사이 → **3번 행**.

여기서 핵심: **offset 배열은 항상 오름차순이다.** 높이는 음수가 될 수 없어 누적값은 무조건 증가한다. 정렬된 배열에서 값의 위치를 찾는 것 → **이진 탐색**.

그래서 선형 탐색 O(n)이 **O(log n)** 으로 바뀐다. 1만 개라도 최대 14번 비교면 끝. 고정 높이의 O(1)보다는 느리지만, 순진한 O(n)과는 비교가 안 된다.

### 장치 3: 게으른 계산 + estimatedItemSize (문제 3 해결)

1만 개 offset을 처음부터 다 계산하면 초기 비용이 크다. 그래서 react-window는 **방문한 데까지만 정확히 계산하고, 안 가본 영역은 추정치(`estimatedItemSize`)로 추측한다.** 스크롤이 진행되며 방문 영역의 offset이 정확히 채워지고, 안 가본 영역은 추정으로 남는다. 그래서 스크롤바 길이가 스크롤하면서 미세하게 조정되기도 한다.

## 4. resetAfterIndex

offset 캐시는 한 번 계산하면 저장해두고 재사용한다. 그런데 5번 행 높이가 나중에 바뀌면(데이터 업데이트로 텍스트가 길어지는 등)? 캐시엔 옛 offset이 그대로 남아 **5번 이후 모든 행 위치가 어긋난다.**

그래서 직접 캐시를 무효화해야 한다.

```jsx
const listRef = useRef();

// 5번 행 높이가 바뀌었을 때
listRef.current.resetAfterIndex(5); // "5번부터 다시 계산해"
```

이게 측정 안 하고 개발자를 믿는다의 대가이다.

**높이가 바뀌면 개발자가 책임지고 캐시를 무효화**해야 한다. → 높이에 대한 책임이 개발자에게 있기 때문. virtuoso는 측정하기 때문에 캐시 무효화가 필요하지 않다.

## 5. 스크롤 setState를 프레임 단위로 묶기 (rAF throttle)

가상 스크롤이든 아니든, 스크롤 핸들러는 1초에 수십~수백 번 호출된다.

매번 `setState`하면 React가 과하게 리렌더링되니, 업데이트를 **1프레임에 한 번**으로 제한(throttle)하는 패턴이 자주 같이 쓰인다. `requestAnimationFrame`(rAF)으로 구현한 훅을 뜯어봤다.

```tsx
export function useFrameValue<T>(initialValue: T) {
  const [value, setValue] = useState(initialValue);
  const frameRef = useRef<number | null>(null); // 예약된 프레임이 있나? (깃발)
  const nextValueRef = useRef(initialValue); // 다음 프레임에 반영할 최신값

  const setFrameValue = useCallback((nextValue: T) => {
    nextValueRef.current = nextValue; // 1 최신값 일단 보관
    if (frameRef.current !== null) return; // 2 이미 예약됐으면 끝
    frameRef.current = requestAnimationFrame(() => {
      // 3 없으면 새로 예약
      setValue(nextValueRef.current); // 4 프레임 도착 시 최신값으로 setState
      frameRef.current = null; // 5 깃발 내림
    });
  }, []);

  useEffect(
    () => () => {
      // 언마운트 시 cleanup
      if (frameRef.current !== null) {
        cancelAnimationFrame(frameRef.current); // 남은 예약 취소
      }
    },
    []
  );

  return [value, setFrameValue] as const;
}
```

### 세 재료의 역할

- `value` — 실제로 화면에 반영되는 state (리렌더 유발).
- `nextValueRef` — "다음 프레임에 반영할 가장 최신 값"을 임시 보관. 값을 **버리지 않기 위한** 저장소.
- `frameRef` — 이 훅의 핵심. "지금 예약된 프레임이 있나 없나"를 기억하는 깃발. `null`이면 예약 없음, 숫자(id)면 예약 중.

### 1+2 조합이 핵심: 값은 살리고, 예약은 한 번만

한 프레임 안에 `10 → 20 → 30 → 40 → 50` 5번 호출됐다고 하면:

```
1번째(10): 예약 없음 → rAF 예약 (③)
2번째(20): nextValueRef=20, 예약 있음 → return
3번째(30): nextValueRef=30, 예약 있음 → return
4번째(40): nextValueRef=40, 예약 있음 → return
5번째(50): nextValueRef=50, 예약 있음 → return
→ 프레임 도착: setValue(50)
```

포인트 두 가지.

1. **값을 버리지 않는다** : 2~5번째는 프레임을 새로 예약하진 않지만 `nextValueRef`는 계속 최신값으로 덮어쓴다. 그래서 프레임이 도착했을 때 마지막 값 50이 반영된다. 중간값(20,30,40)은 건너뛰되 최신값은 절대 놓치지 않는다.
2. **setState는 프레임당 1번 :** 5번 호출돼도 rAF 예약은 1번뿐이라 리렌더도 1번. 이게 throttle 효과다.

### cleanup이 필요한 이유

`() => () => {...}` 구조는 "useEffect에 넘긴 함수가 또 다른 함수를 반환"하는 형태이고, 반환된 함수가 언마운트 시 cleanup으로 실행된다. 스크롤하던 컴포넌트가 사라졌는데 예약된 프레임이 남아 있으면, 프레임이 도착했을 때 이미 사라진 컴포넌트에 `setValue`를 호출하게 된다. 그래서 언마운트 시 남은 예약을 `cancelAnimationFrame`으로 취소한다.

> 정리: `nextValueRef`는 최신값을 계속 들고 있고, `frameRef`는 이번 프레임에 이미 예약했는지를 기억한다. 호출이 아무리 쏟아져도 프레임당 `setValue`는 한 번, 그 한 번에 최신값이 반영된다. 고빈도 스크롤 이벤트에서 리렌더 횟수를 프레임 수준으로 확 줄여준다.

## 6. 고정 vs 가변 정리

| 연산            | FixedSizeList                 | VariableSizeList              |
| --------------- | ----------------------------- | ----------------------------- |
| 위치 계산 (top) | `index × itemSize` — O(1)     | `offset[index]` (캐시) — O(1) |
| 시작 index 탐색 | `scrollTop ÷ itemSize` — O(1) | offset 이진 탐색 — O(log n)   |
| 전체 높이       | `itemCount × itemSize`        | 누적 offset + 추정            |
| 높이 변경 대응  | 불필요                        | `resetAfterIndex` 수동 호출   |

VariableSizeList는 **offset 캐시 + 이진 탐색**이라는 자료구조로, 깨졌던 성능을 거의 고정 높이 수준까지 되살린 셈이다.

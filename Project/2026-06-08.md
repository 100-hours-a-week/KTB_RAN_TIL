# 2026-06-08 일일 회고

## TL;DR

다음 개인 프로젝트의 방향을 확정했다.
**"3D 스크롤 스토리텔링" 영역의 기존 라이브러리를 해부하고, 직접 구현하며 비교하는 프로젝트.**
단순히 라이브러리를 하나 더 만드는 것이 아니라, 기존 구현의 설계 의도를 이해하고 내 손으로 재현해보면서 "왜 이렇게 설계했는가"를 글로 남기는 것이 목표다.

---

## 왜 이 방향인가

### 배경

상반기 지원 결과를 정리하면서, 내가 다음에 채우고 싶은 역량이 무엇인지 다시 생각해봤다.
결론은 **"코드 레벨에서 더 깊은 개발자가 되고 싶다"** 였다.

라이브러리를 _쓸 줄 아는_ 것과 _내부 동작을 이해하는_ 것은 다르다.
지금까지는 좋은 도구를 골라 잘 조합하는 데 집중했다면, 이제는 그 도구가 어떻게 동작하는지, 왜 그렇게 설계되었는지를 설명할 수 있는 사람이 되고 싶다.

### 요즘 고민: "AI로 구현만 했다"는 느낌을 어떻게 벗어날까

AI로 코드 구현이 쉬워진 시대에, "구현했다"는 사실만으로는 더 이상 차별점이 되지 않는다고 느꼈다.
중요한 건 결과물이 아니라 **구현 과정에서 내린 의사결정**이라고 생각한다.

| 표면적인 설명                                       | 깊이가 드러나는 설명                                                                                     |
| --------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| "GSAP ScrollTrigger로 스크롤 애니메이션을 구현했다" | "스크롤 성능을 측정해보니 메인 스레드 블로킹이 있어서, rAF로 throttle하고 will-change를 이렇게 관리했다" |
| "라이브러리를 써서 만들었다"                        | "기존 라이브러리 동작을 뜯어보고, 왜 이렇게 설계했는지 이해한 뒤 직접 구현해 비교했다"                   |

그래서 이 프로젝트는 **측정과 비교를 메인 콘텐츠**로 잡기로 했다.

---

## 컨셉을 정하기까지의 과정

### 1차 아이디어: 감성 인터랙션 컴포넌트 라이브러리

처음에는 react bits 같은 감성 인터랙션 컴포넌트 모음을 만들고 싶었다.
일러스트와 3D를 다뤄본 경험을 살려 "디자인하는 프론트엔드 개발자"라는 포지셔닝을 강화하는 방향이었다.

### 검증: 이미 존재하는가?

방향을 잡기 전에 "내가 만들려는 게 이미 있는가"를 확인했다.

- **react bits**: 스크롤 기반 컴포넌트가 이미 다수 존재 (Animated Content, Scroll Reveal, Scroll Stack). GSAP ScrollTrigger 기반의 2D 효과들.
- **r3f-scroll-rig (14islands)**: DOM 요소와 3D 씬을 동기화하는 성숙한 라이브러리. 내가 막연히 생각하던 "스크롤로 3D 공간을 이동하는" 영역에 가장 가까운 기존 구현.
- **Theatre.js + R3F**, **@react-three/drei의 ScrollControls**: 각각 다른 방식으로 같은 문제를 해결하고 있었다.

즉, "세상에 없는 새로운 라이브러리"라는 전제는 틀렸다. 이미 좋은 구현들이 있었다.

### 결론: 방향 전환

여기서 두 가지 선택지가 있었다.

1. 기존 라이브러리와 차별화되는 완전히 새로운 것을 만든다 → 현실적으로 어렵고, 차별점도 약하다.
2. **기존 라이브러리를 깊이 이해하고, 직접 구현하며 비교한다** → "코드 레벨에서 깊어지고 싶다"는 목표와 정확히 맞는다.

2번을 택했다.
어차피 나는 이 라이브러리들의 동작을 뜯어보고 싶었다. 그렇다면 그 분석 자체를 프로젝트의 중심에 두는 게 맞다고 판단했다.

## 프로젝트 정의

### 한 줄 정의

> 3D 스크롤 스토리텔링 라이브러리의 핵심 동작을 해부하고, 직접 구현해 성능을 비교하는 프로젝트.
> 결과물로 라이브러리와 그것을 사용한 포트폴리오 사이트를 만든다.

### 핵심 원칙

- **분석과 비교가 메인, 결과물은 그 증거물이다.**
- 기존 라이브러리를 전부 뜯지 않는다. **핵심 동작 하나를 골라 끝까지 판다.**
- 모든 개선은 측정으로 증명한다: `가설 → 측정 → 비교 → 개선 → 재측정 → 결론`

### 해부 순서

1. **react bits 스크롤 컴포넌트 (입문)**

   - GSAP ScrollTrigger를 React 환경에서 어떻게 다루는가
   - cleanup, ref 관리, 리렌더 처리 방식 익히기
   - 코드량이 작아 워밍업으로 적합

2. **r3f-scroll-rig (심화)**

   - DOM 스크롤과 3D 씬을 어떻게 동기화하는가
   - 별도 씬을 독립 렌더 패스로 분리하는 구조
   - 핵심 동작 하나만 선택해서 깊게

3. **(참고) Theatre.js + R3F, drei ScrollControls** — 같은 문제의 다른 접근법 비교

## 측정·비교 계획

각 동작마다 "이렇게 했더니 느려서, 이렇게 바꿨다"는 데이터를 남긴다.

| 영역             | 비교 실험                             | 측정 지표          |
| ---------------- | ------------------------------------- | ------------------ |
| 스크롤 throttle  | rAF vs debounce vs raw                | FPS                |
| 3D 객체 정리     | dispose 유무                          | 메모리 누수        |
| 리렌더           | useFrame 내 setState vs ref 직접 조작 | 렌더 횟수          |
| 번들             | 전체 import vs 트리쉐이킹             | 번들 사이즈        |
| 스크롤-3D 동기화 | 기존 라이브러리 방식 vs 내 구현       | 성능 / 코드 복잡도 |

# 2026-06-09 일일 회고

## TL;DR

프로젝트 방향("기존 라이브러리 해부 → 직접 구현 → 비교")을 구체화하기 위해,
**"react bits 같은 복붙형 컴포넌트 라이브러리를 실제로 쓸 때 무엇이 부족한가"** 를 먼저 정리했다.
이 부족한 지점들이 곧 내 라이브러리의 **차별점이자 비교 기준**이 된다.

> ⚠️ 아래 7가지는 아직 "가설" 단계다. 추측으로 두지 않고, 실제로 컴포넌트를 가져와 써보면서 어떤 항목에 부딪히는지 검증할 예정이다. 실제로 부딪힌 것만 차별점으로 삼는다.

---

## 왜 "부족한 점"부터 찾는가

차별점을 처음부터 발명하려 하면 막연하다.
대신 **"기존 걸 실제로 써보면 무엇이 불편한가"** 에서 출발하면, 차별점이 자연스럽게 도출된다.

"같은 컴포넌트인데 이 부분을 이렇게 개선했다"가 있으면, 그게 곧 비교 포인트이고 블로그/자소서 소재가 된다.

---

## react bits류 컴포넌트의 부족할 수 있는 지점 (가설 7가지)

### 1. 복붙 방식이라 버전 관리가 안 됨

react bits는 npm 설치가 아니라 **코드 복붙 방식**이 많다.
한번 가져오면 이후 업데이트·버그픽스를 받을 수 없다.

→ **차별점:** npm 패키지로 설치/업데이트되는 구조

---

### 2. 타입이 부실하거나 `any`

prop 타입이 헐겁게 정의되어 있는 경우가 많다.

→ **차별점:** 엄격한 타입 정의, 자동완성이 잘 되는 API

---

### 3. 커스터마이징이 막힘

애니메이션 속도, easing, 방향 등을 prop으로 바꾸지 못하고
코드를 직접 수정해야 하는 경우가 있다.

→ **차별점:** prop으로 충분히 제어 가능한 설계

---

### 4. 성능 고려 안 됨

스크롤·애니메이션에서 throttle, cleanup, `will-change` 같은 최적화가 빠진 경우.

→ **차별점 (핵심 무기):** 직접 측정해서 최적화했다는 데이터
→ 가설 → 측정 → 비교 → 개선 → 재측정 → 결론 사이클로 증명

---

### 5. 접근성(a11y) 무시

`prefers-reduced-motion` 무시, 키보드 포커스 불가, ARIA 속성 없음 등.

→ **차별점:** 접근성을 챙긴 구현
→ 이전에 Storybook addon-a11y로 색상 대비·접근성 검증을 해본 경험과 연결 가능

---

### 6. SSR / Next.js 호환 문제

`window` 등 브라우저 전역에 직접 접근해서 Next.js SSR 환경에서 터지는 경우.

→ **차별점:** SSR 안전한 구현 (마운트 이후 접근, 가드 처리)

---

### 7. 번들 사이즈

전체를 import해야 해서 무거운 경우. 트리쉐이킹이 안 되는 구조.

→ **차별점:** 트리쉐이킹이 되는 패키지 구조 (`files: ["dist"]`, ESM, sideEffects 설정 등)

## 왜 스크롤인가 (프로젝트의 "왜")

"많고 많은 라이브러리 주제 중 왜 하필 스크롤인가"를 정리했다.
면접에서 반드시 받게 될 질문이고, 억지 이유가 아니라 실제 흐름에서 나온 답이 있어야 한다.

### 1. 스크롤은 사용자가 쓰는 인터페이스 요소 중 하나다

스크롤은 사용자가 웹에서 매 순간 직접 조작하는 요소다.
이 요소 하나를 어떻게 다루느냐에 따라 웹사이트는 **정적인 문서**가 되기도 하고 **살아 움직이는 경험**이 되기도 한다.

스크롤을 내렸을 때 영상이 재생되는 듯한 효과, 3D 공간을 여행하는 듯한 연출 —
react bits 같은 라이브러리가 이런 인터랙션을 잘 보여주고 있고, 스토리텔링은 내가 오래 관심을 가져온 분야였다.
이 스크롤을 잘 살리면 "살아있는 웹사이트"를 만들 수 있겠다고 생각했다.

### 2. 그런데 스크롤은 조금만 끊겨도 경험이 무너진다

화려한 연출은 절반의 이야기다.
스크롤은 사용자가 직접 조작하는 요소라서, 조금만 버벅이면 경험 전체가 무너진다.
그래서 "어떻게 보이게 할까(디자인)"와 "어떻게 끊기지 않게 할까(성능)"는 분리될 수 없는 하나의 질문이다.

→ 이 지점에서 **A(3D 스토리텔링, 디자인)** 와 **B(throttle·가상 스크롤, 성능)** 가 하나로 연결된다.
즉, 둘은 따로 노는 게 아니라 "스크롤이라는 인터페이스 요소를 제대로 이해한다"는 같은 목표의 양면이다.

### 3. 실무에서 이미 스크롤을 다뤘고, 내부가 궁금했다

이전 프로젝트(MOLIP)에서 스크롤 성능 문제를 직접 다뤘다.

- 가상 스크롤 도입 → 렌더 시간 102ms → 19ms (83% 감소)
- ISR + CSR 하이브리드 무한 스크롤 → 중복·누락 0건

그때는 `react-virtuoso` 같은 성숙한 라이브러리를 가져다 썼다.
문제는 해결됐지만 **내부 동작은 블랙박스로 남았다.**
이 프로젝트는 그 블랙박스를 직접 열어보는 데서 출발한다.
라이브러리를 _쓰는_ 개발자에서, 내부를 _이해하고 설명할 수 있는_ 개발자가 되는 것이 목표다.

## "R3F가 취업에 도움이 될까"에 대한 답

프로젝트 주제가 R3F·스크롤이라 타겟 기업(삼성 SDS 사내시스템, 현대, 한화 등)과 안 맞는 게 아닐까 하는 의심이 들었다.
SI·대기업 사내시스템은 데이터 테이블·폼·대시보드가 주력이고 R3F를 쓸 일은 거의 없기 때문이다.

하지만 정리해보면, 기업이 보는 건 **"R3F를 쓸 줄 아느냐"가 아니라 "어떻게 사고하는 개발자냐"** 다.

R3F로 무엇을 만들었는지는 안 봐도, 이건 본다:

- 기존 라이브러리를 뜯어서 이해하는 능력
- 성능을 측정하고 개선하는 능력
- API를 설계하는 능력
- 문제를 정의하고 해결하는 능력

이 역량은 **R3F든 데이터 테이블이든 똑같이 적용된다.**
도구는 핑계고, 사고방식이 핵심이다.

그래서 이 프로젝트는 "R3F를 했다"가 아니라,
**"기존 라이브러리를 분석하고, 성능을 측정·개선하고, API를 설계하고, 문제를 정의해 해결한 과정"** 으로 말해야 한다.
주제가 화려해 보여도 증명하는 역량은 SI 기업이 원하는 것과 동일하다.

> 핵심: 도구(R3F·스크롤)는 표면이고, 증명하려는 건 그 아래의 사고방식이다.

## 역할 분담: 취업 직격 vs 재미·디자인

같은 "스크롤"이라는 주제 안에서, 목적에 따라 결과물의 형태를 나눈다.

| 트랙                  | 내용                                  | 형태                     | 목적                           |
| --------------------- | ------------------------------------- | ------------------------ | ------------------------------ |
| **메인**              | 가상 스크롤 / throttle 직접 구현·비교 | 기술 블로그 + TIL        | 취업 직격 (성능·사고방식 증명) |
| **포트폴리오 사이트** | R3F 스크롤 비주얼                     | 사이트 (라이브러리 데모) | 재미·디자인 감각               |

- **메인(블로그)** 은 SI·대기업이 원하는 "성능을 측정하고 개선하는 사고방식"을 직접 증명하는 트랙이다. throttle·가상 스크롤은 모든 기업이 쓰는 기술이라 직격타다.
- **포트폴리오 사이트** 는 R3F 스크롤 비주얼로 "디자인하는 개발자"라는 정체성과 재미를 챙기는 트랙이다.
- 둘은 분리돼 있지만 "스크롤"이라는 하나의 관심사로 묶인다. 취업과 재미를 양쪽 다 포기하지 않기 위한 구조다.

> 우선순위상 메인(취업 직격)이 먼저고, 포트폴리오 사이트는 그 위에 디자인을 입히는 단계다.

## 컴포넌트 구성 방향 (난이도 그라데이션)

부족한 점들을 개선 기준으로 삼아, 난이도를 나눠 컴포넌트를 구성한다.

```
├─ 쉬움 (2D 범용)  : 텍스트 애니메이션, fade-in, 카운터 등 2~3개
│                    → 기본기 + "개선 감각" 증명 (위 7가지 중 일부 적용)
└─ 어려움 (3D 핵심): 스크롤-3D 씬 1개
                     → 남들이 못 하는 깊이 (r3f-scroll-rig 해부 기반)
```

- 쉬운 컴포넌트라도 **최소 하나의 차별점**(타입/번들/접근성/성능 중)을 적용한다.
- 그냥 따라 만들기(클론)는 의미 없다. "같은 컴포넌트인데 이 부분이 다르다"가 핵심.

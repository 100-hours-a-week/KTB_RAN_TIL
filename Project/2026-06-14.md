# TIL: Node.js는 대체 어디 있고 무슨 일을 할까

> "어차피 브라우저가 JS를 실행하는데 Node는 왜 필요하지?"라는 의문에서 출발해, JS 엔진 · 런타임 · 빌드 · 배포까지 Node의 위치와 역할을 정리한 기록.

---

## Session 0. 왜 이게 궁금해졌나

React 코드의 실행 순서를 따라가다 보니 "JS 엔진", "Node", "브라우저", "바벨", "번들러"가 계속 나왔는데, 정작 이것들이 **서로 어떤 관계인지, 각각 어디 있는지**가 머릿속에서 뒤엉켜 있었다. 특히 이 질문이 풀리지 않았다.

> **브라우저가 JS를 실행하는데, Node.js는 왜 따로 필요하지?**

이 질문 하나를 파고들면서 정리한 내용이다.

---

## Session 1. JS 엔진은 혼자 못 돈다

먼저 큰 그림. **JS 엔진(V8 등)은 JS 코드를 읽고 실행하는 "두뇌"일 뿐이다.** 두뇌만으로는 화면을 그리거나, 파일을 읽거나, 네트워크 요청을 할 수 없다. 그런 일을 하려면 **"손발"**이 필요한데, 그 손발은 엔진 바깥에서 붙여준다.

그 "두뇌 + 손발"을 묶어놓은 게 **런타임(runtime)**이고, 런타임은 크게 두 종류다 — **브라우저**와 **Node.js**.

```
[브라우저]                          [Node.js]
┌─────────────────────┐           ┌─────────────────────┐
│  JS 엔진 (V8)        │           │  JS 엔진 (V8)        │  ← 같은 두뇌!
│  ───────────────    │           │  ───────────────    │
│  + DOM, fetch,      │           │  + 파일시스템(fs),   │
│    window, 이벤트   │           │    http, process    │
│  (브라우저가 제공)   │           │  (Node가 제공)       │
└─────────────────────┘           └─────────────────────┘
   웹페이지 실행용                     서버·도구·빌드용
```

흥미로운 점: **두뇌(V8)는 둘 다 똑같다.** 크롬과 Node.js 모두 구글의 V8 엔진을 쓴다. 차이는 **그 두뇌에 어떤 손발(API)을 붙였느냐**다.

|                 | 브라우저      | Node.js        |
| --------------- | ------------- | -------------- |
| JS 엔진         | V8            | V8 (같음)      |
| DOM, window     | ⭕            | ❌             |
| 파일 시스템(fs) | ❌            | ⭕             |
| 서버(http)      | ❌            | ⭕             |
| 용도            | 웹페이지 실행 | 서버·도구·빌드 |

## Session 2. Node.js란 무엇인가

**Node.js = 브라우저의 JS 엔진(V8)을 떼어내, 브라우저 밖에서 JS를 실행할 수 있게 만든 런타임(실행 환경).**

원래 JavaScript는 브라우저 안에서만 돌아가는 언어였다. 그런데 "브라우저의 V8 엔진을 떼어다 밖에서도 JS를 돌리면 어떨까?" 해서 나온 게 Node다.

핵심은 **DOM/window 대신 파일·서버·시스템 기능을 갖췄다는 것.** 그 덕에 JS로 할 수 있는 일이 확 넓어졌다.

1. **서버 만들기** — JS로 백엔드/API 구축 (Node가 유명해진 이유)
2. **빌드 도구 실행** — 바벨, 번들러(Vite/Webpack)가 Node 위에서 돎
3. **패키지 관리** — npm이 Node에 딸려옴
4. **CLI 도구·스크립트** — 터미널에서 도는 각종 도구

---

## Session 3. 바벨·번들러는 Node "안"에 있는 게 아니다

여기서 흔한 오해 하나를 풀었다. **바벨이나 번들러는 Node에 내장된 게 아니라, Node 위에서 "돌아가는" 별개의 프로그램**이다.

```
Node.js = JS를 실행하는 "환경"(런타임)
   └─ 그 위에서 실행되는 별도 프로그램들 (npm으로 설치):
        ├─ Babel / SWC / esbuild  (JSX·TS 변환)
        ├─ Vite / Webpack         (번들링)
        └─ 기타 도구들
```

비유하면 **Node는 운영체제, 바벨/번들러는 그 위에 설치하는 앱**이다. 윈도우 안에 포토샵이 내장된 게 아니라 윈도우 위에서 포토샵을 실행하는 것처럼.

- **Node에 바벨이 있나?** → 아니다. `npm install`로 따로 깐다.
- **Node가 번들링을 하나?** → 아니다. Node 위에서 도는 번들러가 한다.

단, "Node로 빌드한다"는 말이 틀린 건 아니다. 정확히는 **"Node 위에서 도는 빌드 도구로 빌드한다"**는 뜻.

## Session 4. 왜 번들링은 Node에서만 가능할까

"번들링을 브라우저에서 하면 안 되나?" → **안 된다.** 이유는 번들러가 하는 일을 보면 명확하다.

번들러는:

- 파일 시스템을 뒤져 수십 개 파일을 **읽고**
- node_modules의 수백 개 패키지를 **찾아** 연결하고
- 합쳐서 새 파일을 디스크에 **쓴다**

이 **파일 읽기/쓰기/탐색**이 핵심이다. **브라우저는 보안상 사용자 컴퓨터의 파일 시스템에 함부로 접근하지 못한다.** (악성 사이트가 내 파일을 다 읽으면 큰일이니까) 반면 **Node는 파일 시스템 접근이 자유롭다(`fs`).** 그래서 파일을 묶는 번들링은 Node에서만 가능하다.

```
번들러의 일: 파일 읽고 → 합치고 → 파일 쓰기
   브라우저: 파일 시스템 접근 ❌ (보안 차단)
   Node:    파일 시스템 접근 ⭕ (fs)
   → 번들러는 Node 위에서만 돎
```

> ### 모듈(import)은 사실 브라우저도 이해한다
>
> 헷갈리기 쉬운 점: **모듈(import/export)은 번들링과 다른 얘기**이고, 브라우저도 모듈을 쓸 수 있다.
>
> ```html
> <script type="module">
>   import { hello } from "./hello.js"; // 브라우저가 이거 됨!
> </script>
> ```
>
> 즉 모듈은 Node 전용이 아니다. **번들링만 Node 전용**이다.

## Session 5. 그럼 번들링을 왜 "굳이" 하나

브라우저가 모듈을 이해한다면, 번들링 없이 import 그대로 쓰면 되지 않나? 이론상 되지만 실무에선 문제가 생긴다.

**문제 1 — 요청 폭발**
import가 100개면 브라우저가 파일 100개를 각각 따로 네트워크 요청한다. 그 파일들이 또 import하면 또 요청... 수백~수천 요청이 폭발해 느려진다. 번들링은 이걸 몇 개 파일로 묶어 요청 수를 확 줄인다.

**문제 2 — 브라우저가 못 읽는 것들**

- JSX, TypeScript → 브라우저가 모름 (변환 필요)
- `import _ from 'lodash'` 같은 패키지 이름 import → 브라우저는 "lodash가 어느 파일?"인지 경로를 모름

이런 것들 때문에 **변환 + 경로 해석 + 번들링**이 필요하고, 그걸 Node 위 도구가 해준다.

|                         | 브라우저에서 됨?     | Node 필요? |
| ----------------------- | -------------------- | ---------- |
| 모듈(import) 자체       | ⭕ (type=module)     | ❌         |
| 번들링 (파일 묶기)      | ❌ (파일시스템 불가) | ⭕         |
| JSX/TS 변환             | ❌ (문법 모름)       | ⭕         |
| 패키지 이름 import 해석 | ❌ (경로 모름)       | ⭕         |

## Session 6. 핵심 답 — 브라우저가 실행하는데 왜 Node가 필요한가

이제 처음 질문에 답할 수 있다.

> **브라우저는 "완성된 순수 JS"만 실행한다. 그런데 내가 쓴 React 코드는 완성된 게 아니라 변환·번들링이 필요한 "재료" 상태다.**

내가 쓴 코드를 브라우저에 그대로 주면 에러가 난다.

- `<h1>` (JSX) → 표준 JS가 아니라 브라우저가 모름
- `.tsx` (TypeScript) → 브라우저가 모름
- 수십 개 파일 + 수백 개 패키지 → 그대로 받으면 너무 느림

이 변환·정리 작업을 해주는 도구(바벨/번들러)도 어딘가에서 실행돼야 하는데, 그곳이 Node다.

```
내 React 코드 (JSX, TS, 여러 파일) — 재료
        ↓
   [Node 위에서 바벨/번들러 실행] ← Node의 역할: 요리
        ↓
브라우저가 읽을 수 있는 순수 JS 파일 — 완성품
        ↓
   [브라우저가 실행] ← 식사 (Node 불필요)
```

**Node = "브라우저가 먹을 수 있게 코드를 요리하는 주방."** 브라우저는 완성된 요리를 먹기만 하고, 요리 자체는 Node에서 한다. 그래서 "브라우저가 실행한다"와 "Node가 필요하다"는 충돌이 아니라, **요리(Node) → 식사(브라우저)** 순서로 둘 다 필요한 것이다.

## Session 7. 개발환경 vs 배포환경 — Node는 언제 필요한가

Node가 도는 곳은 "서버"만이 아니다. **개발할 때 내 컴퓨터에서도 Node가 돈다.**

```
① 패키지 설치 — 내 컴퓨터의 Node
   npm install → react, vite 등 다운로드

② 개발 중 — 내 컴퓨터의 Node
   npm run dev → Vite 개발 서버, JSX 즉석 변환, 핫 리로드

③ 빌드 — 내 컴퓨터/빌드 머신의 Node
   npm run build → JSX 변환 + 번들링 → dist 폴더에 결과물

④ 배포 후 — SPA냐 SSR이냐에 따라 갈림 ★
```

### 결정적 차이: SPA vs SSR

```
[SPA] (createBrowserRouter / CSR)
  빌드 후 → 정적 파일(HTML/JS/CSS)만 서버에 올림
  → 브라우저가 받아서 실행
  → 배포 서버에 Node 불필요 ✅
  (변환·번들링이 ③에서 이미 끝났으니 결과물만 있으면 됨)

[SSR] (Next.js 서버 컴포넌트 등)
  빌드 후에도 → 서버의 Node가 계속 돎
  → 사용자 요청마다 실시간으로 HTML 생성 → 응답
  → 배포 서버에 Node 필요 ❌→⭕
```

**개발/빌드 단계에선 항상 Node가 필요하다**(JSX 변환·번들링 때문). 차이는 배포 후다.

- **SPA**: 빌드 결과물만 올리면 끝 → 배포 서버에 Node 불필요
- **SSR**: 빌드 후에도 서버에서 Node가 계속 돌며 요청마다 HTML 생성 → Node 필요

> ### 비유로 정리
>
> - **빌드** = 요리 (재료 손질 → 완성). 항상 주방(Node)에서.
> - **SPA 배포** = 완성된 도시락을 진열대에 올려둔 것. 손님(브라우저)이 가져가기만 하면 됨. 주방 불필요.
> - **SSR 배포** = 주문마다 즉석 요리. 주방(Node)이 계속 돌아야 함.

---

## 한 줄 결론

> **Node.js는 브라우저 밖에서 JS를 실행하는 런타임이다.** 같은 V8 엔진을 쓰되 DOM 대신 파일·서버 기능을 갖췄다. React 개발에서 Node는 **내가 쓴 코드(JSX·TS·여러 파일)를 브라우저가 읽을 수 있는 순수 JS로 변환·번들링하는 작업장** 역할을 한다. 개발/빌드 단계에선 늘 필요하고, 배포 후에는 SPA면 불필요(정적 파일만), SSR이면 필요(요청마다 HTML 생성)하다.
